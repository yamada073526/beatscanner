## 1. v32 → v33 で完了した修正

### 1.1 Fix 1: insights を prefetchAll に追加（根本原因の解消）

**問題箇所**: [api.js](frontend/src/api.js)
```javascript
// 旧（v32 まで）
export function prefetchAll(ticker) {
  fetch(`/api/guidance/${t}/basic`).catch(() => {});
  fetch(`/api/chart/${t}/summary`).catch(() => {});
  // ❌ /api/insights は呼ばれていなかった
}
```

ユーザーが検索ボックスでホバー / 候補表示時に `prefetch()` が走るが、最も重い insights API がスキップされていた。analyze API 完了後に InsightsPanel が mount してから insights API が走る **シリアル動作** だったため、最下層のセクションでスクロール到達時にまだローディング中になっていた。

**修正**: prefetchAll に追加。
```javascript
fetch(`/api/insights/${t}`).catch(() => {});
```

**動作変化**:
- analyze API（5-10秒）と insights API（13-60秒）が**並列実行**に
- ホバー時点から既に insights 取得が開始しているケースが多数
- スクロール到達時のローディング遭遇率が大幅低下

### 1.2 Fix 2: addToWatchlist で insights を fire-and-forget

**問題箇所**: [App.jsx](frontend/src/App.jsx) の `addToWatchlist`

ウォッチリストに追加しても、cron バッチ（毎朝 04:00 JST）まで insights が生成されない。**追加直後にその銘柄を分析しても遅い**。

**修正**:
```javascript
// 既存ロジックに追加
fetch(`/api/insights/${encodeURIComponent(t)}`).catch(() => {});
```

ウォッチリスト追加 → バックグラウンドで Supabase に insights 保存 → 次回閲覧時に Supabase キャッシュからインスタント表示。

### 1.3 Fix 3: スケルトン実コンテンツ形状化 + ProgressSteps

**問題箇所**: [InsightsPanel.jsx](frontend/src/components/InsightsPanel.jsx)
```jsx
// 旧
{loading && [1,2].map(i => <div style={{height:80, ...灰色pulse}} />)}
```

灰色長方形 2 つだけ。最大 60 秒待つケースで **ユーザーが「壊れている」と判断** する典型例。

**修正**: 2 つの新コンポーネントを追加:

#### A. `InsightsSkeleton` — 実コンテンツ形状
```
┌─────────────────────────────┐
│ ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒  │ ← summary 3行 (100/92/70%幅)
│ ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒    │
│ ▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒      │
├──────────────┬──────────────┤
│ 🟢 強気材料  │ 🔴 弱気材料  │
│ ▒▒▒▒▒       │ ▒▒▒▒▒        │
│ ▒▒▒▒▒       │ ▒▒▒▒▒        │
│ ▒▒▒▒▒       │ ▒▒▒▒▒        │
└──────────────┴──────────────┘
```
シアン基調のスケルトン（強気側）+ 赤基調（弱気側）で本来の見た目を予告。

#### B. `ProgressSteps` — 楽観的タイマーで進捗演出
```
✓ ニュースを収集            (3秒経過で点灯)
⟳ AI で統合分析中           (8秒経過で✓化)
○ 強気・弱気材料を抽出      (15秒経過で✓化)

ℹ️ 初めて分析する銘柄のため、通常より時間がかかっています（最大60秒）
📌 次回からは即座に表示されます  (15秒以降に表示)
```
- バックエンドからの SSE は不要、**フロントの予測タイマー**で十分「進んでる感」を演出
- 15 秒以降は「マイナー銘柄は時間がかかる」案内で**長く待っても許容される心理**を作る

#### 適用箇所
- 未ログインユーザー向けビュー（`if (!user)` 内）
- ログイン済ユーザー向けビュー（メイン return 内）
両方のローディング表示を統一。

### 1.4 Fix 4: 判定タブのセクション再編成

**問題**: 最も engaging な「市場の声」が最下層に埋もれていた。

#### 変更前
```
1. ResultBadge (PASS/FAIL)
2. ConditionCard × 5
3. 「AI詳細レポートを見る」ボタン
4. GuidanceCard
5. HistoryChart
6. StockPriceChart
7. IRLinksPanel
8. NewsPanel
9. InsightsPanel ⭐ ← 最下層に埋没
```

#### 変更後（v33）
```

2. ConditionCard × 5
3. 「AI詳細レポートを見る」ボタン
4. GuidanceCard

──[ 市場コンテキスト ]──        ← 🆕 セクション区切り（シアン縁）
```

InsightsPanel と NewsPanel を「市場コンテキスト」として隣接配置（両方とも市場の文脈情報のため）。

### 1.5 Fix 5: セキュリティヘッダー追加

**問題箇所**: [main.py](backend/app/main.py) のミドルウェア

CORS 設定はあったが、**XSS / クリックジャッキング / リファラ漏洩対策のヘッダーが未設定**。

**修正**: CORS の直後に新規ミドルウェアを追加。
```python
@app.middleware("http")
async def add_security_headers(request: Request, call_next):
    response = await call_next(request)
    response.headers["X-Content-Type-Options"] = "nosniff"
    response.headers["X-Frame-Options"] = "DENY"
    response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
    return response
```

| ヘッダー | 効果 |
|---|---|
| `X-Content-Type-Options: nosniff` | MIME sniffing 攻撃を抑止 |
| `X-Frame-Options: DENY` | iframe 埋め込み禁止（クリックジャッキング対策） |
| `Referrer-Policy: strict-origin-when-cross-origin` | クロスオリジン時にパス情報を漏らさない |

**未対応（v34 以降）**:
- `Content-Security-Policy` — Stripe + Google OAuth + Supabase の通信制約を考慮した設計が必要

### 1.6 Fix 6: モバイルタブにラベル復活

**問題箇所**: [App.jsx](frontend/src/App.jsx) のフローティングボトムナビ
```jsx
// 旧
{!isMobile && <span>{tab.label}</span>}  // モバイル時は非表示
```

モバイルではアイコンのみで「📊 / 📋 / 📈」が並ぶ → **初見ユーザーが何のタブか分からない**。

**修正**:
```jsx
<span style={{ fontSize: isMobile ? 18 : 16 }}>{tab.icon}</span>
<span style={{ fontSize: isMobile ? 10 : 12 }}>{tab.label}</span>
```

タブボタン自体も `52→60px` 幅 / `44→48px` 高に拡張してラベル領域を確保（Apple HIG の 44px 推奨も維持）。

## 7. v33 で確立した永続ルール候補

CLAUDE.md への昇格を検討すべき項目:

### 7-1. プリフェッチ運用
- 重い API（Claude / RSS 系）は **必ず prefetchAll に含める**
- ユーザー操作（ホバー / ウォッチリスト追加）の時点で fire-and-forget 起動
- バックエンドのキャッシュ層に依存しすぎない（フロント側からも温める）

### 7-2. ローディング UI 設計
- スケルトンは **実コンテンツの形状を予告** すること（灰色ブロックは NG）
- 5 秒以上かかる可能性がある操作は **進捗ステップ** を用意（楽観的タイマーで OK）
- 15 秒以上のケースには **「初回は時間がかかる」案内** で心理的負担軽減

### 7-3. 情報設計の原則
- 「ナラティブピラミッド」: 結論 → 視覚 → 根拠 → ストーリー → 詳細・参照
- 最も engaging なコンテンツを上半分に
- 外部誘導（IRリンク等）は最下層

### 7-4. セキュリティ最低ライン
- 全レスポンスに `X-Content-Type-Options` / `X-Frame-Options` / `Referrer-Policy`
- 新エンドポイント追加時は CORS allow_methods に追加するか確認
