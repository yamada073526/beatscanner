## 5. 触ると危険な箇所 (CLAUDE.md + v69 + v70 追加)

### v70 追加 (= 今日の dogfood で発見・確定した教訓)

#### a. Modified Dietz `start_shares` は `period_from - 1day` を使う
- backend `/api/portfolio-performance` の Modified Dietz 計算
- 旧仕様 (`start_shares = _shares_at(period_from)`) は **期間内 buy が cashflow と評価額で 2 重控除** されて P/L 反転
- memory `feedback_modified_dietz_period_open.md` 参照
- 新規 period 系 endpoint (例: `/api/portfolio-history`) でも同 ロジック必須

#### b. Chip primitive 規律 (round 7-9 で SSOT 確定)
- 全 chip 系 UI は `<Chip>` primitive 経由でしか描画しない (inline style + `.ds-chip` 直書き禁止)
- 新規 add-action は `<Chip variant="add">` を使う (Plus icon + dashed border)
- rollup と個別 item の階層分離は `<ChipSeparator />` を使う (`role="rollup"` + Separator)
- memory `chip_primitive_canonical.md` 参照
- design-system-check skill が違反検査 (`elevation_scale.md` の ALLOWED-CHIP / ALLOWED-PLUS-ICON whitelist)

#### c. `ChipGroup.Separator` (compound pattern) は Vite minify で tree-shake される
- `Component.Subcomponent = ...` の side-effect assignment は Vite minify で保証されない
- 必ず **named export** で `<ChipSeparator />` のように書く
- React compound component pattern を BeatScanner で使うときは要注意

#### d. prop drilling で内側 component に渡し忘れると React tree crash
- 外側 component (PortfolioHoldingsList) の signature 変更 → 内側 component (HoldingRowCompact) の関数 scope に変数なし → ReferenceError → 真っ白
- formatter / hook を内側で呼ぶときは **signature 全段で props を受け取る**
- 段階的再導入で 1 component ずつ確認すると即座に発見可能

#### e. 画面真っ白の最速 diagnostic = AI サマリーに debug append
- backend の response shape を変えず、 ai_summary 末尾に `[v2:cp=N/M ...]` のような debug 情報を 1 行 append
- 本番 user (= ログイン session 必要) でしか再現できない bug の primary 診断手段
- Sentry / DevTools より速い、 ユーザーに「console error スクショ」を依頼するより速い
- 4 段階で絞り込み (overall → ticker 別 → aggregate → cashflow 別) で根本原因特定可能
