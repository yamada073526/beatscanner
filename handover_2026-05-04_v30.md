### 1.1 FMP APIキー Supabase クラウド同期

#### 背景
従来の `fmpKey.js` は localStorage 専用。デバイス間でAPIキーが共有できなかった。

#### 実装内容
- `frontend/src/lib/fmpKey.js` に3つの async 関数を追加(既存4関数は維持)
  - `saveFmpKey(key, supabase)` — localStorage + Supabase の両方に保存
  - `loadFmpKey(supabase)` — Supabase優先、失敗時は localStorage フォールバック
  - `clearFmpKey(supabase)` — 両方からクリア
- **mirror戦略**: `loadFmpKey` が Supabase 値を localStorage にミラーする → 以後の同期 getter は localStorage で動作(オフライン対応)
- `ApiKeySettings.jsx` に「☁ クラウドに同期済み」シアンバッジ(保存後3秒表示→フェードアウト)
- `App.jsx` に `[authReady, user]` 依存の useEffect で起動時・ログイン変化時に自動ロード

#### Supabase テーブル(手動作成済み)
```sql
create table user_settings (
  user_id uuid references auth.users primary key,
  fmp_api_key text,
  updated_at timestamptz default now()
);
alter table user_settings enable row level security;
create policy "Users can manage own settings"
  on user_settings for all
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);
grant all on public.user_settings to authenticated;
```

#### 設計判断のポイント
- `getFmpKey` は `api.js` から毎 HTTP リクエスト時に**同期で**呼ばれるため async 化は破壊的
- 既存の同期 getter を維持しつつ、新規 async 関数を並列で追加する設計を採用
- Supabase クライアントのパスは `../lib/supabase.js`(指示書の `supabaseClient` は誤り)

#### 🆕 補強: 同期/非同期 API の役割分担表
| 関数 | 種別 | 用途 | 呼び出し元 |
|---|---|---|---|
| `getFmpKey()` | sync | ヘッダー値生成 | `api.js`(毎リクエスト) |
| `setFmpKey(key)` | sync | localStorage 即時更新 | `saveFmpKey/clearFmpKey` 内部 |
| `hasFmpKey()` | sync | UI 表示判定 | `planGating.js`, `App.jsx` |
| `getMaskedKey()` | sync | マスク表示 (`****XXXX`) | `ApiKeySettings.jsx` |
| `saveFmpKey(key, supabase)` | async | 検証後の保存 | `ApiKeySettings.jsx` |
| `loadFmpKey(supabase)` | async | 起動時 + ログイン変化時 | `App.jsx` useEffect |
| `clearFmpKey(supabase)` | async | 削除ボタン | `ApiKeySettings.jsx` |

### 1.2 市場の声(InsightsPanel) — じっちゃまナレッジ統合

#### 背景と設計方針
- structured.md(投資解説ライブの構造化テキスト)をナレッジベースとして活用
- **"じっちゃま" という表記はUI・コード・コメントに一切出さない**
- 表示名: 「市場の声」(中立的な表現)
- 複数ソース対応を前提とした設計(将来的にStockTraders Almanac等も追加可能)

#### ファイル構成
```
backend/data/insights/   ← structured.md 14本を手動配置
  2019-08-30_full.md
  2026-03-15_full.md
  ...(14本)
  2026-05-01_full.md
```

ファイルの手動コピーコマンド:
```bash
cp ~/Projects/investment-knowledge-base/transcripts/structured/lives/*.md \
   ~/Projects/beatscanner/backend/data/insights/
```

```

② Supabase market_insights(24h以内)
```

#### APIレスポンス形式
```json
{
  "ticker": "NVDA",
  "overall_sentiment": "negative",
  "summary": "統合見解テキスト(400字)",
```

#### トラブル履歴と解決策(重要・今後に活かせる知見)

| 試行 | 問題 | 解決策 |
|---|---|---|
| 第1回 | 503 "Knowledge base not available" | `.gitignore` が `railway up` のアップロードをブロック → コメント化で解決 |
| 第2回 | 500 "prompt is too long: 223090 > 200000 tokens" | 14ファイル全体は200K超 → **直近6ファイル制限**(`_INSIGHTS_MAX_FILES = 6`) |
| 第3回 | 500 "Analysis parse error" — Claude がナレッジベース内容をそのままecho back | **3層防御**: ①system promptに「最初の文字は{」明記 ②参考資料を`<knowledge_base>`XMLタグで包む ③末尾に再指示 |
| 第4回 | まだ JSONパース失敗 | **assistant prefill `{`** を追加(ClaudeClient に `prefill` 引数を追加)→ 構造的にJSON以外を返せなくなる |
| **完了** | ✅ HTTP 200 / found: true / 4 mentions | 3層防御+prefillの組み合わせで解決 |

#### 教訓
- **Claude に JSON のみを返させる最強の方法は assistant prefill `{`**
- XML タグで参考資料を包むと「これは出力ではなく参照対象」の強いシグナルになる
- プロンプトの末尾は特に影響力が高いため、末尾で再指示することが重要

#### 🆕 補強: タイムアウト多層防御(永久ローディング防止)
NXT 等マイナー銘柄で永遠ローディングが発生した問題への対策:

| 層 | 場所 | 値 |
|---|---|---|
| RSS 1ソース fetch timeout | [rss_collector.py: `fetch_rss(timeout=15)`](backend/app/rss_collector.py) | 15秒 |
| BE 全体のオンデマンド処理 | [main.py: `asyncio.wait_for(_refresh_one(tkr), timeout=60.0)`](backend/app/main.py) | 60秒 |
| FE fetch | [InsightsPanel.jsx: `AbortController` + `setTimeout(75000)`](frontend/src/components/InsightsPanel.jsx) | 75秒 |
| BE タイムアウト時の挙動 | `asyncio.TimeoutError` を捕捉 → `found:false` レスポンスをキャッシュ保存(連打抑制) | — |

**設計**: BE 60秒 < FE 75秒 で安全マージン。タイムアウト時は必ず `found:false` を返し、フロントは「準備しています」+「🔄 もう一度分析する」ボタンを表示。

### 1.3 RSS自動収集パイプライン(Phase 2a)

#### 構成
- `backend/app/rss_collector.py`(新規) — Yahoo Finance + Seeking Alpha RSS収集
- `backend/app/main.py` — `/api/insights/refresh/{ticker}` と `/api/insights/refresh/batch` エンドポイント
- `railway.toml` — `[[crons]]` で毎日 UTC 19:00 = JST 04:00 に自動実行

#### Railway Cron の設定
```toml
[[crons]]
schedule = "0 19 * * *"
command = "curl -X POST https://beatscanner-production.up.railway.app/api/insights/refresh/batch -H 'X-Cron-Secret: $CRON_SECRET'"
```

🆕 **Cron 時刻変更履歴**: 当初 `"0 21 * * *"` (UTC 21:00 = JST 06:00) で設定 → 翌日にユーザー指示で `"0 19 * * *"` (UTC 19:00 = JST 04:00) に前倒し変更。前場開始前のデータ準備が目的。

#### Supabase テーブル(手動作成・GRANT実行済み)
```sql
create table market_insights (
  id uuid default gen_random_uuid() primary key,
  ticker text not null,
  overall_sentiment text,
  summary text,
  bull_points jsonb default '[]',
  bear_points jsonb default '[]',
  key_metrics jsonb default '[]',
  sources jsonb default '[]',
  collected_at timestamptz default now(),
  updated_at timestamptz default now()
);
create unique index idx_market_insights_ticker_unique on market_insights(ticker);
alter table market_insights enable row level security;
create policy "Authenticated users can read insights"
  on market_insights for select using (auth.role() = 'authenticated');
create policy "Service role can manage insights"
  on market_insights for all using (auth.role() = 'service_role');
-- 以下も実行済み(GRANTを忘れるとservice_roleでも書き込めない)
GRANT SELECT, INSERT, UPDATE, DELETE ON public.market_insights TO service_role;
GRANT USAGE ON SCHEMA public TO service_role;
GRANT SELECT ON public.watchlist TO service_role;
```

#### 🆕 補強: パフォーマンス実測値
- 1銘柄 RSS+Claude+Supabase upsert: **約 13〜16秒**
- batch (7銘柄): **約 78〜100秒**
- batch は **sequential 処理**(並列ではない) — 20銘柄超なら paging 必要

#### トラブル履歴と解決策(重要・今後に活かせる知見)

| 問題 | 原因 | 解決策 |
|---|---|---|
| `POST /api/insights/refresh/batch` が `{"ticker":"BATCH","status":"no_news"}` を返す | FastAPI はルート登録順に評価。`/refresh/{ticker}` が先に登録されていたため `batch` が ticker として処理された | **`batch` エンドポイントを `{ticker}` の前に登録**(ルート順序は厳守) |
| Supabase upsert が `42501 permission denied` | Supabase新バージョンでは新規テーブルに service_role への GRANT が自動付与されない | 明示的に `GRANT SELECT, INSERT, UPDATE, DELETE ON public.market_insights TO service_role` を実行 |
| CRON_SECRET が途中で切れて見える(42文字に見えたが実際は64文字) | Railway CLI の table 表示幅が約40文字で切れる | `railway variables --json` で完全な値を取得する |
| 🆕 watchlist テーブルへの service_role GRANT も別途必要 | 上記 GRANT は market_insights のみ。batch は watchlist 読み取りも必要 | `GRANT SELECT ON public.watchlist TO service_role` を追加実行 |

### 1.4 禁止語サニタイザ(3層防御)

#### 背景
structured.md に「じっちゃまが〜と言っていた」という記述があり、Claude APIがそれを要約する際に「氏は〜」「アナリストによると〜」等の個人主語が混入した。

#### 3層防御の構造

| 層 | 実装 | カバー率 |
|---|---|---|
| ① system prompt 初版 | 「氏」「アナリスト」等の主語禁止ルールを明記 | 73% |
| ② system prompt 強化版 | 所有格・目的語も禁止。❌→✅の変換例5パターン拡張 | 91% |
| ③ 後処理サニタイザ(Python) | `_sanitize_insights_text/data` で機械的に置換(70パターン以上) | **100%** |

#### サニタイザの適用箇所
- `_analyze_text_to_insights()` — Claude出力直後
- `get_insights()` — Supabase行返却時(既存データも遡及適用)

#### 🆕 補強: サニタイザの置換パターン詳細
[main.py: `_sanitize_insights_text()`](backend/app/main.py) に約70パターンを定義。順序重要(長いパターンから先に置換)。

| カテゴリ | 例 (置換前 → 置換後) |
|---|---|
| 投資家(複合語) | `機関投資家` → `機関`, `個人投資家` → `個人` |
| 投資家(所有格) | `投資家の信頼感` → `市場での信頼感`, `投資家の主要な` → `市場の主要な` |
| 投資家(主語/目的語) | `投資家は` → `市場では`, `投資家へ` → `市場へ` |
| アナリスト | `アナリストによると` → `市場では`, `アナリストの` → `市場の` |
| 専門家/市場参加者/ストラテジスト | `専門家の` → `市場の`, `市場参加者は` → `市場では` |
| 氏/彼/自身 | `氏自身` → `(削除)`, `氏の` → `市場の`, `彼の` → `その` |

**重要**: Supabase に既に保存されている古いデータも `get_insights()` から返す前にサニタイズが効くため、テーブルを上書きしなくてもユーザー側には clean データのみ届く。

#### 教訓
- **プロンプトだけでは Claude の出力を100%制御できない**
- 特に「投資家の主要な懸念材料」のような所有格パターンはプロンプトで防ぎにくい
- 重要な制約(禁止語・出力形式)は必ずPython側で後処理サニタイザを設けること

### 1.5 検索ボックス UX改善

#### 問題と解決

| 問題 | 原因 | 解決策 |
|---|---|---|
| TSLA がオートコンプリートに出ない | FMP APIが「TSLA」で検索するとETF説明文のTSLAを返し、本物のTSLA(Tesla)が出なかった | `/api/search` に MASTER_TICKERS 優先マッチングを前段追加 |
| ボックス外クリック後、↓キーで候補が再表示されない | `onBlur` で suggestions がクリアされており、再フォーカス時に空の状態 | 検索ロジックを `fetchAndOpen()` ヘルパーに切り出し。`onFocus` と `ArrowDown` から再フェッチ |
| 1Passwordのバナーが検索ボックスに被る | input が `type="text"` でパスワードマネージャーが介入 | `type="search"` + `autocomplete="off"` 等の組み合わせで解消 |

#### 🆕 補強: MASTER_TICKERS 優先マッチングの実装詳細
[main.py: `/api/search`](backend/app/main.py) — FMP/yfinance タスク起動の直後に挿入:

1. **完全一致** (`q.upper() == ticker.upper()`) → 必ず先頭
2. **prefix マッチ**(完全一致以外) → 最大 5 件
3. 既存 FMP + yfinance 結果を重複排除 (シンボル) して merge
4. exchange は `"NASDAQ"` 固定(マスタ由来は概ね US 主要市場、具体名は FMP/yf データで上書き)
5. 社名は既存の `_TICKER_NAMES` 辞書から取得 (TSLA → "Tesla Inc." 等)
6. `source: "master"` フィールドで識別可能

#### 🆕 補強: input に追加した4属性
[TickerSearch.jsx](frontend/src/components/TickerSearch.jsx) の input 要素:
```jsx
<input
  type="search"            // ← パスワードマネージャー判別の最強シグナル
  autoComplete="off"
  autoCorrect="off"        // iOS 自動修正抑制
  autoCapitalize="off"     // 自動大文字化抑制
  spellCheck={false}       // スペルチェック抑制
  // name / id 属性は意図的に付けない (1Password の識別子手がかりを与えない)
  ...
/>
```

#### 🆕 補強: `fetchAndOpen()` 共通ヘルパー
`searchTickers` 呼び出しを debounce 内から関数に切り出し、3箇所で再利用:
- `handleChange` の debounce 内 (250ms)
- `onFocus` (即時、suggestions が空なら fetch)
- `handleKeyDown` の `ArrowDown && !open` 分岐 (即時)

#### 教訓
- `type="search"` は見た目の変化が少ないが、パスワードマネージャーへのシグナルとして有効
- `data-1p-ignore="true"` や `data-lpignore="true"` という次の手もある(現状は不要)
- name/id 属性を付けないこと自体が 1Password 識別子の手がかりを与えない有効な対策

## 6. 既知の技術的制約・地雷

### 触ってはいけない箇所(v29から継続)

| 箇所 | 理由 |
|---|---|
| sticky 検索バー | 8回試行錯誤の末確定 |
| ヘッダー grid 構造 | mobile: 2カラム / md+: 3カラム |
| タブキー | `'home'/'judgment'/'report'/'チャート'`(最後だけ日本語) |
| ニュースホバー perspective設定 | `:has()` は廃止済み。JS制御に変更済み |
| `scroll-reveal` に `!important` を付けない | ニュースカードのホバー演出が死ぬ |

### 市場の声 運用上の注意

| 注意点 | 詳細 |
|---|---|
| ナレッジベースの更新 | `backend/data/insights/` は `.gitignore` 対象外として運用。新しいファイルを追加したら手動コピー → `railway up` |
| 直近6ファイル制限 | 全14ファイルは250K tokens で200K上限超過。`_INSIGHTS_MAX_FILES = 6` で制限中 |
| バッチ最大20銘柄 | ウォッチリストが20超えたら paging が必要 |
| Yahoo Finance RSS の将来廃止リスク | Finnhub(`FINNHUB_API_KEY` 既設定済み)を代替として組み込み可能 |
| 🆕 Cron 検証 | 翌朝 04:00 JST 後に Supabase で `SELECT updated_at FROM market_insights ORDER BY updated_at DESC` で確認 |
| 🆕 銘柄追加時 | watchlist に新規銘柄を追加すると、次回 batch (翌朝 04:00) から自動的に対象。即座に分析したければ手動 `POST /api/insights/refresh/{ticker}` |

### Supabase の落とし穴

| 落とし穴 | 対策 |
|---|---|
| 新規テーブルに service_role GRANT が自動付与されない | 必ず `GRANT SELECT, INSERT, UPDATE, DELETE ON public.{table} TO service_role;` を実行 |
| upsert に `on_conflict` を指定する場合は UNIQUE 制約が必要 | インデックス作成時に `unique` を付ける |
| 🆕 watchlist テーブルの service_role GRANT も別途必要 | batch エンドポイントで watchlist 読み取り時に必須 |
| 🆕 `railway up` は `.gitignore` を尊重する | 本番に含めたいファイル/ディレクトリは .gitignore から外す必要あり |

### FastAPI のルート順序

static path(`/batch` 等)は必ずワイルドカード(`/{ticker}`)より**前に**登録すること。
後に登録すると `batch` が `ticker="BATCH"` として処理される。

## 10. v30 で学んだ落とし穴まとめ(今後に活かせる知見)

| 落とし穴 | 状況 | 解決策 |
|---|---|---|
| Claude が JSON でなくナレッジベース内容をecho back | system prompt だけでは不十分 | assistant prefill `{` + XMLタグ + 末尾再指示の3層防御 |
| プロンプトで禁止語を100%除去できない | 「投資家の主要な懸念材料」等の所有格が残存 | Python側で後処理サニタイザ(70パターン)を実装 |
| Railway CLI が長い変数値を途中で切る | CRON_SECRET が42文字に見えたが実際は64文字 | `railway variables --json` で完全な値を取得 |
| Supabase 新規テーブルに service_role GRANT が自動付与されない | upsert で `42501 permission denied` | 明示的に GRANT を実行する(テーブル作成後の必須手順) |
| FastAPI のルート順序問題 | `/refresh/batch` が `/refresh/{ticker}` に吸収される | static path を先に登録 |
| `.gitignore` が `railway up` のアップロードをブロック | data/insights/ が本番環境に届かない | `.gitignore` をコメント化(railway up は .gitignore を尊重する) |
| 250K tokens で Claude API の上限超過 | 14ファイル全体を投げると失敗 | 直近6ファイル制限で135K tokensに抑制 |
| `type="text"` の input に1Passwordが介入 | 検索ボックスにパスワードバナーが被る | `type="search"` + `autocomplete="off"` 等の組み合わせ |
| 🆕 watchlist テーブルへの service_role GRANT 漏れ | batch endpoint が watchlist 読めず空を返す | `GRANT SELECT ON public.watchlist TO service_role` 追加 |
| 🆕 マイナー銘柄の永久ローディング | RSS が極端に遅い/0件 | BE `asyncio.wait_for(60s)` + FE `AbortController(75s)` 多層タイムアウト |
| 🆕 Supabase 既存データ内の禁止語が GET レスポンスに残る | 古い行が 24h 内なら採用される | `get_insights()` の Supabase 経路でも sanitizer を適用 |
