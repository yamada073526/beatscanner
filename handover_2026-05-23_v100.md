- **VITE_ARG/ENV 同期** (Dockerfile Stage 1)

- **FMP `/stable/stock-peers` の response 形式変更**
- **FMP 部分確定 data**: 最新四半期で 1 segment のみ確定 → 「2+ segments 揃った Q」 選択ロジック必須
- **vision-eval frame 8 統一** (G-4 で対策済)
- **ProfileCard sanitize 漏れ pattern**

### v100 新規教訓
- **clipping bug は wrapper chain で発生** ([[feedback-clipping-root-cause-chain]] と同 pattern): DetailReport の `paddingLeft 12` のみ (右 padding 無し) + AccordionSection panelInner 左右 padding 無しの両者が combined して「左右クリッピング」 user 認識を発生。 1 wrapper 修正で消えなければ全 wrapper を grep で trace + symmetric padding に変更
- **mobile flex layout で flexWrap 無し = character break**: Pane4Inspector header が iPad 横で 6 子要素 1 行詰込み → wid 圧縮 → span が 1ch まで圧縮 → 文字単位 wrap (「13」「時」「間」「前」 縦並び)。 outer + inner flex 両方に `flexWrap: 'wrap'` + 各 inline label に `whiteSpace: 'nowrap'` で完全防止
- **native `title=` attribute は touch device で発動しない**: chart 凡例 ⓘ chip 等の help tooltip は click popover で実装すべき。 useState + outside click close + role="dialog" + design token (--bg-card/--border/--shadow-md) で実装
- **markdown rendering の dark mode 対応必須**: tailwind `text-slate-*` 固定 class は dark mode で contrast 不足。 var(--text-primary/secondary) + var(--color-accent) で token 化、 h2 15px / h3 13px / strong cyan / list 20px indent + 4px gap で hierarchy 明確化
- **vision-eval NVDA UNRELIABLE bug 真因**: `button:has-text(ticker)` broad locator + post-click assert 無しで watchlist 外 ticker (NVDA 等) → modal を採点。 v100 で post-click `[data-testid="pane3-hero"] h1` assert + fail-fast (exit 3) で完全防止

## 3. iOS 版 path multi-review 3 体合議 verdict (2026-05-23 前半)

SPEC: [docs/specs/SPEC_2026-05-23_ios-path-roadmap.md](docs/specs/SPEC_2026-05-23_ios-path-roadmap.md)

### UI designer verdict — **C 案 (bottom nav + drawer) 推奨**

- 5 tab (案 A) / 4 tab (案 B) は情報密度均等化で hierarchy 喪失、 mobile では「賑やかだが迷う」
- **C 案 bottom nav 3 軸 (ホーム / 銘柄 / ポートフォリオ) + drawer (ニュース / scanner)** が「メイン柱 2-3 本」 + minimalism 原則と一致
- **情報密度**: mobile は意図的に減らすべき。 first fold は TriageBanner + 判定サマリー + portfolio balance の 3 要素に絞る (Aman ロビーとヴィラのシーン分離 §-1-B)
- **Phase 2 (token 統一)**: delaying value でなく土台投資、 C 案再構築時の cost 40-50% 削減。 ただし **token / color / typography のみに限定**、 nav 変更は C 案に一本化
- **代替 (将来)**: full-screen card stack + swipe (Robinhood 方式) は「驚き」 感情語彙最強だが実装 cost C 案 1.5-2x、 Phase 4 Next.js 移行後に温存

### Frontend architect verdict — **真因 = cross-component event miss + GRANT 抜け落ち複合**

- **真因 hypothesis (最有力)**: `useTransactions` の `loadedForUserRef` skip 条件 + `bs:transactions:changed` broadcast が cross-component で届かない設計。 旧 UI で `TransactionEntryModal` → `addTransaction` → broadcast、 同一 page mount 済 component にしか届かない → mobile 旧 UI 入力時に workspace 側 `PortfolioDetailBody` は未 mount で listener なし、 次回 workspace 開いた時 `loadedForUserRef.current === user.id` が既 set で useEffect skip → **mount 時の one-shot fetch のみで cross-session invalidation 発生せず**
- **次に疑う**: v80/v84 で再発した service_role GRANT 抜け落ち (transactions/accounts/forex_rates の SELECT 権限) が新 UI 側だけ silently empty を返す pattern
- **C 案 vs D 案**: portfolio modal 系は旧 UI 固有 state 多い → 同 component で `md:` breakpoint 分岐より **旧 UI 全体を mobile shell wrapper する方式** が安全。 D 案 (Next.js) は記事タブまで凍結 CLAUDE.md 方針と一致、 **C 案先行**

#### audit 順序 (Phase 0 で実行、 Frontend architect 提示)
1. browser console で `fetchTransactions` 手動実行 → rows 返るか確認 (RLS + GRANT 最短切り分け)
2. `useTransactions` の `loadedForUserRef` skip 条件: workspace 開いた直後の `reload()` 起動を React DevTools で確認 (cross-session stale 再現)
3. `bs:transactions:changed` event listener: 旧 UI で transaction 登録後、 workspace tab で `PortfolioDetailBody` mount + listener bind 確認
4. backend `/api/portfolio-judgment` を `curl + jq` で直接叩き、 `{"detail": ...}` エラー or 正常 JSON 確認 (service_role GRANT server-side 切り分け)
5. Supabase SQL Editor で `select grantee, privilege_type from information_schema.role_table_grants where table_name in ('transactions','accounts') and grantee = 'service_role'` — SELECT 不在なら v84 `v84_service_role_grants.sql` 相当を再適用

### QA dogfooder verdict — **Phase 1 release 前 mandatory + Trust Cliff 該当**

- **Trust Cliff 判定**: 該当する可能性大。 LP 訴求「保有銘柄も判定」 vs 旧 UI 入力 → 新 UI 非反映 = 典型的 Trust Cliff bug。 現状 dogfood 段階のため CVR 即時ダメージ 0 だが「動いていない機能を release する」 こと自体が brand 信頼の地雷
- **Phase 0 audit (0.5-1 人日) を先行**: 真因特定だけなら小工数、 release 前にやる価値
- **Phase 1 fix (3-5 人日)**: release 前 mandatory、 Phase 0 + Phase 1 セットで release 前完了
- **Phase 2 (旧 UI token 統一)**: release 後 1-4 週以内。 B 案継続なら旧 UI が design token 乖離を月単位で蓄積、 6 ヶ月後「2 コードベース実質メンテ」 状態。 brand 統一 cost は 2-3 人日で抑えられる今が安い
- **Phase 3 (mobile workspace 再構築)**: release 後 2-3 ヶ月以内に **mobile user share 計測 (GA4/Sentry UA tag) で判定**。 mobile 比率 30%+ なら C 案着手、 30% 未満なら D 案 (Next.js responsive) と統合して先送り可
- **mobile user share 測定基盤** は release 直後で OK。 GA4 `device_category` tag は既存 gtag.js に `event_params` 追加のみ 30-60 分。 Sentry `user_agent` tag は init 1 行追加。 release 前に工数を使う必要なし

### 3 体合議統合 verdict

1. **Phase 0 audit (0.5-1 人日)** → release 前 mandatory、 Frontend architect の audit checkpoint 1-5 順に実行
2. **Phase 1 portfolio 整合性 fix (3-5 人日)** → release 前 mandatory、 真因に応じて (a) `loadedForUserRef` skip 削除 + 外部 event listener 拡張、 (b) service_role GRANT 再適用、 (c) cross-session event broadcast → React Context or Supabase Realtime に置換
3. **Phase 2 旧 UI token 統一 (2-3 人日)** → release 後 1-4 週、 token / color / typography のみ。 nav 変更は C 案へ一本化
4. **Phase 3 mobile workspace 再構築 (C 案 bottom nav + drawer)** → release 後 2-3 ヶ月以内に mobile user share 30%+ なら着手
5. **Phase 4 Next.js 移行** → 記事タブ launch 前後で D 案、 Robinhood 方式 (full-screen card stack) はその後の拡張候補
6. **mobile user share 測定基盤** → release 直後 30-60 分で実装

→ **release 前 mandatory: Phase 0+1 (合計 3.5-6 人日)**
→ **release 後 priority: Phase 2 (1-4 週) → mobile share 測定 → 30%+ なら Phase 3 (2-3 ヶ月)**
→ **長期: Phase 4 Next.js 移行 (記事タブ launch 前後)**
