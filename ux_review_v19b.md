# UXレビュー beatscanner v19b

レビュー日時: 2026-04-29
対象バージョン: main ブランチ最新（operatingMargins/valuation/bullCase/bearCase 追加後）

---

## 🔴 重大（すぐ直すべき）

### 1. スマホでグロースチャートが右側切れ（コンテンツクリッピング）

**問題説明:**
スマホ（375px幅）で「数字で見る成長ストーリー」パネルを表示すると、2列×2行（800px固定幅 SVG）のグラフ右半分がビューポート外に溢れる。右側パネル（CFPS, 営業CF）の棒グラフ・数値ラベルが視認できない。スクロールヒント（「← スクロールして全体を確認」）は表示されるが、ユーザーがそれに気づく前に読み飛ばしてしまう恐れが高い。

**原因:**
`buildGrowthSVG()` が `viewBox="0 0 800 ..."` で2列固定レイアウトを生成し、コンテナ側で `minWidth: '720px'` を指定している（DetailReport.jsx L504-506）。スマホ幅では横スクロールが必要になるが、ユーザーはコンテンツが途切れているとは気づきにくい。

**改善案:**
- `buildGrowthSVG()` にスマホ向けの1列レイアウト分岐を追加する。具体的には `window.innerWidth < 640` のとき `rows=4, cols=1` で縦スタックのSVGを生成する（`SVG_W = 400, PANEL_W = 380`）。
- あるいは各グラフパネルをSVGではなくReactコンポーネント（Recharts等）で実装し直し、レスポンシブに対応させる。
- 最低限の暫定対応として、横スクロールコンテナの上部に固定表示される視認性の高いアイコン付きインジケーター（例: 「→ スワイプで右を確認」バッジ）をスマホ時のみ表示する。

### 2. ビジネスモデルフローボックスの英単語途中折り返しバグ

**問題説明:**
"Microsoft Cloud が牽引" のような英日混在テキストがフローボックス内（`BOX_W=136px`）に収まらず、"Microsof / t Cloudが / 牽引" のように英単語の途中で改行される。スクリーンショット（Image 4）で確認済み。ブランド名・商品名が崩れると信頼性を損なう。

**原因:**
`tspanWrap()` の英単語境界判定（DetailReport.jsx L55-87）は「非英数字→英数字境界で切る」ロジックを持っているが、`maxCharsPerLine=8` という厳しい制限に対して英単語が長すぎる場合に `wordEnd <= maxCharsPerLine + 8` の条件（L75）を超えてしまい、フォールバックとして `cut = maxCharsPerLine` の強制截断が走る。

**改善案:**
- `tspanWrap()` の `maxCharsPerLine + 8` の余裕を `maxCharsPerLine + 16` に拡大し、長い英単語を1行に収める。
- フローボックスの `detail` テキスト（subline）は現在 `maxCharsPerLine=8` だが、`BOX_W=136` でフォントサイズ11pxならば9〜10文字程度は収まるため値を9に上げる。
- 根本対応：AIプロンプト側で `detail` フィールドを8文字以内の日本語に限定する旨の制約を加える（英単語が混入しにくくなる）。

### 3. スクロールホバーエフェクトがスマホで誤発火（パフォーマンス・視覚ノイズ）

**問題説明:**
`App.jsx` L71-100 のスクロールイベントハンドラが `document.querySelectorAll('.panel-card')` に対して `transform: translateY(-6px)` と `filter: drop-shadow(...)` を適用している。スマホではスクロール中にすべての `.panel-card` が浮いたり光ったりする誤作動が発生し、視覚的にうるさく操作感を損なう。また `will-change: transform, filter` を全カードに適用しているためGPUメモリも圧迫する。

**原因:**
ホバーエフェクトをスクロールイベントで代替実装しているが、スマホ（hover: none）環境でも動作してしまっている。`index.css` のメディアクエリ分岐はCSS hoverのみで、JSスクロールハンドラへの分岐がない。

**改善案:**
- `App.jsx` のスクロールハンドラに `if (window.matchMedia('(hover: none)').matches) return;` の早期リターンを追加し、タッチデバイスでは完全無効化する。
- `will-change` プロパティを `@media (hover: hover)` スコープに限定するか、ホバー中のみ動的付与する。

## 🟡 中程度（次スプリント）

### 4. VizPanel の強み・リスク列グリッドがスマホで `window.innerWidth` 参照（SSR非対応・タイミング問題）

**問題説明:**
`VizPanel` の強み・リスク対比セクション（DetailReport.jsx L514）と Bull/Bear グリッド（L551）で `gridTemplateColumns: window.innerWidth < 480 ? '1fr' : '1fr 1fr'` を直書きしている。コンポーネントのレンダリング時点でウィンドウ幅が取得されるため、回転（Portrait ↔ Landscape）後に再レンダリングが発生しないとレイアウトが更新されない。

**原因:**
React の状態やCSSメディアクエリを使わず `window.innerWidth` をインライン三項演算子で読んでいる。リサイズイベントやorientation changeに追従できない。

**改善案:**
- `useWindowWidth()` フックを作成して `useState`+`useEffect`+`resize`イベントで幅を管理する。
- あるいはCSS Grid の `auto-fit, minmax(240px, 1fr)` で完全CSSレスポンシブに置き換える（もっともシンプルな対応）。

---

### 5. タブバーのテキストが 4 項目で均等割り付けされ、スマホではタップ可能領域が狭い

**問題説明:**
タブバー（App.jsx L548-595）は `flex-1` で4等分しているため、375px幅では各タブ約86pxになる。絵文字 + 日本語2〜3文字の組み合わせで、実質のタップターゲットは高さ `py-3`（48px相当）あるが横幅が狭い。また「🔒 決算」タブはPro限定ゲート付きで、タップするとアップグレードモーダルが開く挙動がユーザーにとって予測できない。

**原因:**
テキストサイズは `text-sm`（14px）で絵文字込みで内容が詰まっており、さらにプロゲートがある項目への視覚的事前告知（ロックアイコン以外）がない。

**改善案:**
- タブ高さを最低 `min-h-[52px]` に固定し縦方向のタップ余白を確保する。
- 「🔒 決算」タブはホームタブ内または判定タブのCTAとして配置し直し、メインナビから外す（認知コストを削減）。
- タブラベルを絵文字のみにするか、絵文字を小さくしてテキストを優先する。

---

### 6. ヘッドライン（28px bold）がスマホ幅でテキスト折り返し

**問題説明:**
VizPanel のヘッドライン（DetailReport.jsx L435: `fontSize: '28px', fontWeight: '900'`）はAIが生成した15〜20文字程度の日本語文字列だが、スマホでは2行に折り返されてしまう。スクリーンショット（Image 5）で FAIL 銘柄の「回復基調でEPS急伸も利益率が課題」は画面幅に収まっているが、長めの銘柄では問題が出る。

**原因:**
フォントサイズが固定28pxで、コンテナ幅（padding込みで約343px）に対して余裕がない。

**改善案:**
- `clamp(18px, 5vw, 28px)` または `fontSize: 'clamp(18px, 5vw, 28px)'` に変更してビューポート幅に追従させる。
- AIプロンプト側でヘッドライン文字数を18文字以内に制限する制約を追加する。

---

### 7. 成長チャートの Y 軸ラベルが最大値のみ（スケール感が伝わらない）

**問題説明:**
`buildGrowthSVG()` の各パネルでは Y軸の最大値（`maxV`）のみを左上に小さく表示している（DetailReport.jsx L189: `font-size="10" fill="#9ca3af"`）。ゼロ基準線（axisY）はあるが中間のグリッド線・ラベルがないため、棒の高さから絶対値や増減率を直感的に読み取れない。

**原因:**
SVG手書きで中間グリッドを省略している。コード量を抑えた最小実装の結果。

**改善案:**
- Y軸に3〜4本のグリッド線（最大値・75%・50%・25%）とラベルを追加する。
- 各棒の上部の数値ラベル（現在も実装済み: L156）フォントサイズを11px → 12pxに拡大し、視認性を高める。
- 最終棒（最新期、現在は `#38BDF8`）と過去棒（`#cbd5e1`）の差分パーセントを最終棒の脇に `+xx%` 表示する。

### 8. 「ビジュアル分析を生成する」ボタンが決算タブを開くたびに手動アクションが必要

**問題説明:**
決算タブ（DetailReport）を開いた際に、AIテキスト詳報は自動ストリーミング開始されるが、ビジュアル分析は「ビジュアル分析を生成する」ボタンを押すまで生成されない。モバイルユーザーは詳報テキストをスクロールし終えた後にボタンを探す必要があり、生成に気づかないケースがある。

**原因:**
設計上の意図的な2ステップ（テキスト詳報 → 図解は別リクエスト）だが、UXフローとしては滑らかでない。API呼び出し2本のコスト節約と速度優先の設計。

**改善案:**
- AI詳報のストリーミング完了（`done` 状態）後、2〜3秒のdisplayCountdown付きで「次は図解を生成します（キャンセル可）」ポップアップを表示する。
- または詳報エリア下部に「図解を見る↓」アンカーリンクを追加し、生成済みの場合はすぐスクロール、未生成の場合はトリガーするようにする。

### 9. 判定タブの ConditionCard がダークモードで `bg-white` 固定のまま

**問題説明:**
ConditionCard（L324）は `className="panel-card ... bg-white"` とTailwindクラスで白背景を指定している。`index.css` L66 の `[data-theme="dark"] .bg-white { background-color: var(--bg-card) !important; }` でオーバーライドされているが、borderが `border-2 border-pass/40` または `border-fail/40` として直書きされており、ダークモードのborder変数（`--border`）が反映されない。

**原因:**
Tailwindの `border-pass/40`・`border-fail/40` カスタムカラーに対してダークモードオーバーライドCSS変数が定義されていない。

**改善案:**
- `border-2` + カスタムカラークラスの代わりに `style={{ border: `2px solid ${passed ? 'rgba(34,197,94,0.4)' : 'rgba(239,68,68,0.4)'}` }}` でインライン指定に統一する（ダークモード分岐不要）。

### 10. CalendarPanel・IRLinksPanel の情報が銘柄未選択時にも常時表示されており、初期画面が重い

**問題説明:**
アプリ初回表示時（銘柄未選択）でも CalendarPanel（決算カレンダー）・スクリーナー・MoversCard が常時ロードされている。LCP（Largest Contentful Paint）を悪化させる可能性があるほか、モバイルでは初期スクロール量が多くなり、主要CTA（ティッカー検索）への到達が遅れる。

**原因:**
App.jsx のレイアウト上、タブ外にCalendar/Screenerボタンが常時表示・常時レンダリングされている。

**改善案:**
- CalendarPanel・ScreenerPanel は遅延レンダリング（`React.lazy` + Suspense）またはユーザーがボタンを押すまでマウントしない設計に変更する。
- モバイルではスクロール位置に応じたIntersection Observer based lazy loadを導入する。

## 🟢 良い点（維持すべき）

- **AI要約の色分けシステムが直感的:** [POS]/[NEG]/[NEU] タグによる緑・赤・グレーの3色分類はスキャナビリティが高く、決算の良し悪しが2秒で判別できる。デザインガイドの「2秒でメンタルモデルが構築できる」哲学を体現している。
- **PASS/FAIL バッジの視認性が高い:** 丸アイコン（✓/✕）とカラーコーディングが条件カードに一貫しており、5条件のクリア/未クリアが一覧で把握できる。
- **ダークモード実装の完成度:** CSS変数（`--bg-primary` 等）とTailwindの `[data-theme="dark"]` オーバーライドで主要コンポーネントのダーク表示が整合している。夜間利用での目への負荷が低い。
- **Sparklineの活用:** ConditionCard のスパークラインが数値推移をコンパクトに可視化しており、テキスト数値だけでは見逃しやすいトレンドを補完している。
- **スケルトンローディングの設計:** SummaryBrief のシマーアニメーション骨格（skel-line, skel-pulse）がコンテンツ到着前の layout shift を抑制し、知覚速度を向上させている。
- **条件説明モーダルの教育的価値:** ConditionCard の「？」ボタンからアクセスできる詳細モーダルがじっちゃまプロトコルの根拠を丁寧に説明しており、投資初心者の学習体験を高めている。これはみんかぶには存在しない差別化ポイント。
- **営業利益率オーバーレイの追加:** 売上高チャートに黄色の折れ線でマージン推移を重ねる設計（`marginOverlay`）は、みんかぶのグラフを意識した改善であり収益の質の変化が一目でわかる。
- **スマホ向けスクロール入場アニメーション:** MoversCard の ticker pill・keyword の段階的入場演出（`mover-ticker-pill` → `mover-keyword` 0.12s 遅れ）がモバイルでの読み込み感を良くしている。
- **SVGダウンロード機能:** `buildDownloadSVG()` による分析結果のSVG書き出しはSNSシェアやメモ保存に使えるユニークな機能。他の金融情報サービスにはほとんどない。
- **ビジネスモデルフロー図の視覚訴求力:** 矢印つき水色ボックスのフロー図はMSFTのような複合事業の全体像を1枚で伝えられ、テキスト詳報との相乗効果が高い。

## みんかぶ比較：beatscanner の優位点 / 不足点

### 優位点

- **プロトコル準拠の判定ロジック:** みんかぶは財務データの羅列（売上高・営業利益・純利益・EPS・ROE・ROA）を提供するが、じっちゃまプロトコルの「5条件」という明確な投資判定フレームワークに基づいてPASS/FAILを即時判定できる。投資家がデータを読んで「で、買うべきか？」という問いに直接答える設計はbeatscanner独自の強み。
- **AIによるナラティブ生成:** みんかぶのAI分析（ページ最下部）は短いテキスト段落だが、beatscanner は「ビジネスモデルフロー + 成長グラフ + 強みリスク + 投資家への問い + Bull/Bear ケース」という構造化されたビジュアルストーリーを生成できる。コンテキスト密度が圧倒的に高い。
- **リアルタイムストリーミング要約:** 結果到着と同時に要約文字がストリームで流れるUI体験は、みんかぶの静的テキスト表示に比べて生成感・信頼感を演出する。
- **決算カンファレンスコール分析:** ConferenceAnalysis コンポーネントによるCC要点のAI要約はみんかぶに存在しない機能。機関投資家目線のインサイトを個人投資家に提供している。
- **ウォッチリスト + 決算カレンダー連携:** 登録銘柄の決算日をカレンダー上でハイライトする機能はみんかぶにもあるが、beatscanner は分析フローとシームレスに接続されている。
- **CFPS（キャッシュフロー）中心の評価軸:** みんかぶはCFPS/営業CF を前面に出さないが、beatscanner はじっちゃまプロトコルに従い現金創出力を最重要指標として扱う。粉飾リスク検出の観点から投資の質が高い。

### 不足点

- **データテーブル（数値の網羅性）:** みんかぶは5年分の売上高・営業利益・経常利益・純利益・EPS・ROE・ROAをテーブルで提供する。beatscanner は ConditionCard と HistoryChart でトレンドは見せているが、スプレッドシート的にコピー可能な数値テーブルがない。
- **複数期間グラフ切替:** みんかぶは四半期/通期の棒グラフを3パネル並列表示する。beatscanner の growthSVG は最新データ中心で過去の期間数が少なく（通常4〜5本）、長期トレンドの確認が難しい。
- **PER/PBR の業種比較:** みんかぶはバリュエーション指標に「業種平均バッジ」を付与し相対評価が直感的にわかる。beatscanner の valuation セクション（VizPanel L454-482）はPER/PBR/PSRの絶対値と「割安/適正/割高」判定はあるが、業種比較がない。
- **配当・自社株買い履歴グラフ:** みんかぶは配当推移グラフを独立したセクションで提供する。beatscanner は `dividend.yield` と `dividend.buyback` を数値表示のみで可視化できていない。
- **アナリスト予想との乖離（サプライズ率）:** みんかぶはEPS予想値をテーブルに含める。beatscanner はFMP無料プランの制限でEPS/Revenueのサプライズ率が `unknown` になることが多く、Beat/Miss 判定の精度が低い（CLAUDE.md に既知の制限として記載）。
- **モバイル対応の成熟度:** みんかぶはスマホ表示でグラフを縦スタックに再配置しており、クリッピングが発生しない。beatscanner の固定幅SVGグラフはスマホで横スクロールを強いる（上記🔴問題1に該当）。
- **キャッシュフロー計算書セクション:** みんかぶは営業CF・投資CF・財務CFの3分類を棒グラフで提供する。beatscanner はCFPS単一指標での評価に留まりフリーCF・投資CFの内訳が見えない。
- **株価ファンダメンタル連動性の深堀り（セクター/マクロ文脈）:** みんかぶはセクター動向・指数比較リンクがある。beatscanner は個別銘柄分析に特化しており市場全体文脈との接続が弱い。
