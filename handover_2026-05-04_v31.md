## 1. v30 → v31 で完了した実装

### 1.1 Stripe 決済 — Backend

#### 追加した依存関係
[backend/requirements.txt](backend/requirements.txt) に `stripe>=10.0` を追加。

#### 追加したエンドポイント

| エンドポイント | 説明 |
|---|---|
| `POST /api/stripe/checkout` | Supabase JWT 認証必須。`{plan: 'monthly' \| 'yearly'}` を受けて Stripe Checkout Session を作成し URL を返す。月払いは7日間無料トライアル付き。 |
| `POST /api/stripe/webhook` | Stripe 署名検証後にサブスク状態を Supabase `subscriptions` テーブルに反映。`checkout.session.completed` / `customer.subscription.updated` / `customer.subscription.deleted` の3イベントを処理。 |

#### JWT 検証の落とし穴と解決策（重要・今後に活かせる知見）

| 試行 | 問題 | 解決策 |
|---|---|---|
| 第1回 | `sb.auth.admin.get_user(token)` が `'SyncGoTrueAdminAPI' object has no attribute 'get_user'` | `auth.admin` には `get_user` が存在しない（admin API の体系が異なる: `get_user_by_id` のみ） |
| 第2回 | service role クライアントで JWT を検証する方法がない | **anon key Supabase クライアントを別途作成**して `client.auth.get_user(jwt)` を呼ぶ |
| **完了** | ✅ 正常な JWT エラー → 認証通過 | `_get_supabase_anon()` ヘルパーを追加（遅延初期化＋キャッシュ） |

#### Supabase クライアントの2系統運用
| クライアント | 用途 | キー |
|---|---|---|
| `_get_supabase_service()` | RLS バイパス書き込み（market_insights, subscriptions upsert） | `SUPABASE_SERVICE_ROLE_KEY` |
| `_get_supabase_anon()` | JWT 検証専用（`auth.get_user(jwt)`） | `SUPABASE_ANON_KEY` または `VITE_SUPABASE_ANON_KEY` |

#### Stripe Customer の管理
- 既存ユーザーは `subscriptions` テーブルから `stripe_customer_id` を取得して再利用
- 初回ユーザーは `stripe.Customer.create(email, metadata={'supabase_user_id': uid})` で新規作成
- `metadata.supabase_user_id` を経由して Webhook 受信時にユーザーを特定

#### Webhook の Supabase 反映ロジック
| イベント | 処理 |
|---|---|
| `checkout.session.completed` | Stripe Subscription を取得してプラン判定（price ID 比較）→ subscriptions テーブルに upsert（status, plan, trial_end, current_period_end） |
| `customer.subscription.updated` | customer_id から user_id を引いて subscriptions を update（プラン変更や trial→active 移行に対応） |
| `customer.subscription.deleted` | status を `'canceled'` に更新 |

```javascript
// App.jsx:171
const isProUser = isPro() || isSubscribed;
//                  ^^^^^^^   ^^^^^^^^^^^^
```

### 1.3 デプロイ運用の変更（**今後の標準**）

#### 旧運用（v30 まで）
- **`railway up` のみ**を使用（CLAUDE.md でルール化）
- ローカルファイル直送のため Git 履歴と本番が乖離するリスク

#### 新運用（v31 以降）
- **Git push → Railway 自動デプロイ**を標準化
- GitHub: https://github.com/yamada073526/beatscanner
- main ブランチへの push が Railway の Git 連携で自動的にビルド・デプロイをトリガー

#### 切り替え理由（重要）
本セッション中に以下の事故が発生:
- ユーザーが Railway ダッシュボードの "Deploy" ボタンを押した結果、Git ベースのデプロイが走り、`railway up` でアップロード済みのローカル最新コード（Stripe 実装途中）が**古い main の内容で上書き**された
- 復旧のため再度 `railway up` を実行する手間が発生

→ **Git にコミットされていないコードが本番に存在する状態**が事故の原因。常に Git に push してからデプロイする運用に移行。

#### 🆕 新運用での手順
```bash
# 1. 変更を加える
# 2. ローカルでビルド確認
cd frontend && npm run build

# 3. コミット
git add <files>
git commit -m "feat: ..."

# 4. push（→ Railway が自動デプロイ）
git push origin main

# 5. デプロイ完了確認
curl https://beatscanner-production.up.railway.app/
# バンドルハッシュの変化で確認（フロント変更時）
# または curl /api/insights/NVDA 等で API 動作確認
```

#### CLAUDE.md の更新（次セッション着手時の TODO）
v31 完了時点では CLAUDE.md の永続ルールはまだ「`railway up` のみ」のままになっている。次セッションで以下に書き換えること:

## 3. Supabase テーブル構成（v31 時点）

```
watchlist           — ユーザーのウォッチリスト（既存）
user_settings       — FMP APIキー（クラウド同期）（既存）
market_insights     — 市場の声データ（v30 で追加）
🆕 subscriptions     — Stripe サブスク状態（v31 で追加）
```

#### subscriptions テーブル定義
```sql
create table subscriptions (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users not null,
  stripe_customer_id text,
  stripe_subscription_id text,
  status text default 'inactive',  -- 'trialing' | 'active' | 'canceled' | 'inactive'
  plan text default 'free',         -- 'free' | 'monthly' | 'yearly'
  trial_end timestamptz,
  current_period_end timestamptz,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);
alter table subscriptions enable row level security;
create policy "Users can read own subscription"
  on subscriptions for select using (auth.uid() = user_id);
create policy "Service role can manage subscriptions"
  on subscriptions for all using (auth.role() = 'service_role');
GRANT SELECT, INSERT, UPDATE, DELETE ON public.subscriptions TO service_role;
GRANT USAGE ON SCHEMA public TO service_role;
```

## 6. v31 で学んだ落とし穴まとめ（今後に活かせる知見）

| 落とし穴 | 状況 | 解決策 |
|---|---|---|
| `sb.auth.admin.get_user(jwt)` が存在しない | `'SyncGoTrueAdminAPI' object has no attribute 'get_user'` | anon クライアントの `client.auth.get_user(jwt)` を使う |
| Railway ダッシュボードの Deploy ボタンが Git の古い main を本番に展開 | `railway up` 済みの最新コードが上書きされた | Git push → 自動デプロイ運用に移行（v31 以降の標準） |
| Vite のバンドルハッシュは内容ベース | 同じフロントコードなら異なるデプロイでもハッシュが変わらない | バックエンドのみの変更時はハッシュで判定できない → API レスポンスで判定 |
| Docker キャッシュが Python レイヤを再利用してバックエンド更新が反映されない | `railway up` してもエンドポイントの挙動が古いまま | Dockerfile の `COPY backend/` 直前に cache-bust コメントを追加 |
| Stripe `customer.subscription.updated` の price 判定 | サブスク内の `items.data[0].price.id` を `STRIPE_YEARLY_PRICE_ID` と比較してプラン判定 | Webhook でも同じロジックを書く必要がある |
