## 2. 🔴 数字一致の確認ポイント (本番デプロイ後にユーザーに確認頂きたい)

### 2-1. チャート部とリスト部の数字一致

ユーザー指摘の問題例:
- GOOGL を 150 株、avg_cost = $196.33、trade_date = 2026-05-06 (一昨日) で入力
- 旧: チャート部 −0.01% (TWR、close ベース) / リスト部 +104.1% (avg_cost ベース)
- **新 v50**: 両方とも **+104.1% に一致** (累積リターン = (現在評価額 − 累積投下資本) / 累積投下資本)

### 2-2. drift 警告 chip の表示

trade_date 当日の市場終値 ($400) と user_avg_cost ($196.33) が **51% 乖離** しているため、
チャート部の凡例右に **「⚠ 取得単価と購入日が乖離 (1 件)」** amber chip が表示される。
hover で詳細 tooltip: `GOOGL: 取得単価 $196.33 は 2026-05-06 終値 $400.12 と 51% 乖離`。

### 2-3. 法務免責文

ポートフォリオ画面末尾に薄いグレー枠で:
> ※ 表示の数値は参考値です。配当・税金・売買手数料は含まれません。
> 実際の損益は証券会社の取引履歴をご確認ください。
> 本サービスは投資助言ではなく情報提供を目的としています。

## 3. 🟡 重要知見 (v50 セッション固有)

### 3-1. cashflow ロジックは `cost_basis_method` で分岐
- **`user_input` (default)**: `cashflow = shares × user_avg_cost` → 累積リターンと含み損益が一致 (Robinhood / 楽天 / SBI 流)
- **`market_close`**: `cashflow = shares × close(trade_date)` → strict TWR (将来 Premium 機能)
- **`unknown`**: `market_close` と同じ挙動 (購入日不明 lot のフォールバック)
- price 欠落時は `market_close` に自動降格 (旧 frontend 互換)

### 3-2. backend が Single Source of Truth
- 旧: frontend `computeTWR` で幾何連鎖 → backend cashflow に依存して暗黙の前提が分散
- 新: backend が `total_return_pct` を直接返す → frontend は plot するだけ
- 計算式の真実は `backend/app/main.py:1235-1245` のみ。
- frontend `computeTWR` は後方互換のため残るが、新しい backend `total_return_pct` を優先採用。

### 3-3. drift 警告は ±3 営業日 fallback で精度確保
- trade_date 当日の close が無い場合 (休場日等)、±1, ±2, ±3 日の close を線形検索。
- 5% 超で warning 配列に追加、UI で amber chip。

### 3-4. holding_lots スキーマは現状そのまま
- `cost_basis_method` カラムは **Phase 2 で追加予定** (DB migration)。
- 現状は payload に `'user_input'` を frontend が常に送る → 既存ユーザーに影響ゼロ。
- DB 移行 SQL は Phase 2 着手時に handover に記載。
