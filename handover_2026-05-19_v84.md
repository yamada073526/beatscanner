### v84 dogfood loop で確立した 4 つの永続教訓

1. **Dead code 削除時 hook import dependency check** (新 memory `feedback_dead_code_hook_dependency.md`)
   - 同ファイル多 component の場合、 import 削除前に grep 必須
   - build pass は runtime error を捕まえない、 lazy chunk 真っ白事故の典型 pattern

2. **Supabase 新規テーブルは authenticated + service_role 両方に GRANT 必須** (memory `feedback_supabase_grant_bug.md` 更新)
   - v68 grants migration の不備で v82 Phase 5 実装が silent 失敗継続
   - backend `print()` warning は HTTP 200 で返すため frontend は気付かない
   - 確認 SQL: `select * from information_schema.role_table_grants where grantee='service_role'`

3. **signal_quality envelope の `empty` は失敗ではない**
   - frontend の `confidence !== 'high'` 条件で chip 表示 → 「データ無し = 失敗」 と user 誤誘導
   - chip 条件は `hasFatal` (error/timeout 実在) で判定すべき
   - backend の confidence 計算 (`ok_count >= 3` で high) と frontend の chip 表示判定は分離

4. **新 component 実装時の schema fixture check**
   - v82 Phase 5 で書いた `aggregator/triage.py` が transactions の legacy schema (side/qty) を前提
   - test fixture も legacy schema で書かれて pytest pass だが、 本番 DB は正本 schema (type/shares)
   - schema SSOT memory (`portfolio_account_schema.md`) を読まずに新 code を書くのは禁忌
