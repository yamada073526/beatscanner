# Tool Reliability Prevention — ツール呼び出しテキスト化の再発予防

Claude Code セッション中に **ツール呼び出しが実行されず、ハーネス内部のツール呼び出し構文がそのまま地の文テキストとして出力される** 不具合（以下「テキスト化」）の予防・検知・再発時 runbook をまとめた SSOT。

- 初出: 前々セッション (handover v171 → v172 移行時) で頻発、v172 セッションでも再発。
- v172 は line 11 で「詳細は末尾『## tool reliability』」と予告したが、**その節自体が書き残せず失われた**（テキスト化と同根の可能性）。本 doc がその正式版。
- ⚠️ **本 doc 自身が context に読み戻されたとき再発の呼び水にならないよう、完全なツール呼び出しサンプルは意図的に貼らない**（理由は §予防策 P2）。

> これは **beatscanner のコードバグではない**。Claude Code ハーネス／モデル層の挙動であり、リポジトリのコード修正では直らない。直せるのは「運用」と「context 衛生」のみ。

---

## 症状 (これが出たら本 doc)

- ツール呼び出しが実行されず、`antml` prefix 付きの invoke タグ等の **XML 様構文がそのまま assistant の地の文として transcript に表示される**。
- prefix の脱落（v172 の典型「antml prefix 脱落」）/ タグの入れ子崩れ / 開始タグだけ出て結果が返らない、など形は様々。
- 「ツールを呼んだはずなのに **結果ブロックが返ってこない / 何の効果も起きていない**」のは、ほぼテキスト化を疑ってよい。
- 一度出ると **同一セッション内で再発しやすくなる**（後述の priming）。

---

## なぜ起きるか (寄与要因 — 確度順・仮説含む)

ハーネスは特定の構造化フォーマットで書かれたツール呼び出しブロックだけを tool-use イベントに変換する。フォーマットが少しでも崩れると parse に失敗し、fallback として **ただのテキスト** にレンダリングされる。崩れを誘発する要因（上ほど user/Claude が制御でき、効果が大きい）:

1. **Context priming（最重要・制御可能）** — context 内にツール呼び出し構文に似たテキストが既に存在すると、モデルがそれを「テキストとして」模倣しやすくなる。対象は2種類:
   - (a) **直前にテキスト化した壊れたツール呼び出し**そのもの（だから同一セッションで雪だるま式に悪化する）。
   - (b) **完全なツール呼び出しサンプルを埋め込んだファイル／handover／コメント／doc** が読み戻されたとき。← 本 doc がサンプルを伏せる理由。
   - ※ (b) は (a) ほど確証は強くないが、低コストな予防として「貼らない」を徹底する。
2. **Context 圧 / 長大な単一 turn** — 生成が長くなる・context 占有が増えるほど構造化フォーマット遵守が劣化する傾向。**1M context セッションは特に曝露が大きい**。
3. **複雑な multi-tool メッセージ** — 1 メッセージに多数の並列ツール呼び出しや深い入れ子を詰め込むと崩れ面が増える。
4. **長い narrative の直後にツール呼び出し** — 大量の地の文を吐いた末尾でツールを呼ぶ形は崩れやすい。
5. **(確度低) transient なハーネス／ストリーミング描画** — モデルでなく描画層起因のケース。これは運用で防げない。

## 予防策 (今すぐできる運用 SOP)

| # | 策 | 効く要因 | 状態 |
|---|---|---|---|
| **P0** | **壊れた出力を次セッションに持ち込まない** — テキスト化した断片を新セッションや handover に貼らない | 1(a) | ✅ 本移行で実施済（user 明言「壊れた出力は持ち込んでいません」） |
| **P1** | **再発し始めたら早めにセッション移行** — 同一セッション内で雪だるま式に悪化するので「粘って直そうとしない」 | 1(a),2 | v172 で実証（移行で収束） |
| **P2** | **完全なツール呼び出し構文をファイル/handover/comment/doc に書かない** — 必要なら構文を伏せる・プレースホルダ・抽象記述で。本 doc 自身も遵守 | 1(b) | 本 doc で確立 |
| **P3** | **terse / atomic ツール運用** — 不安定時は 1 メッセージ 1 ツール、preamble（呼び出し前の地の文）を最小化 | 3,4 | ✅ v172 で切替 |
| **P4** | **context を軽く保つ** — handover は `/fetch-handover` の圧縮 summary のみ、memory は MEMORY.md index 行のみ（既存 lazy read SOP と一致） | 2 | 既存 SOP（CLAUDE.md「コスト効率運用」5-6） |
| **P5** | **長大な単一 turn を分割** — 巨大な reasoning + 多数ツールを 1 turn に詰めない。フェーズを切る | 2,4 | — |
| **P6** | **fragile 時は並列ツール数を絞る** — 平常時は独立呼び出しの並列化は推奨だが、テキスト化が出ている間は直列に落とす | 3 | — |
| **P7** | **重い探索をサブエージェントに委譲** — サブエージェントは独立 context window で動き main context を汚染しない。大量 grep / ファイル探索 / ログ解析は委譲して main の context 圧を直接下げる（`model: sonnet` 併用で cost も最適、既存コスト SOP と一致） | 2 | 平常時から設計に組込 |

> P3〜P6 は **「テキスト化が観測されている間」のフェイルセーフ**。平常時まで常に atomic にする必要はない（並列ツール呼び出しは平常時は効率的）。**症状が出たら締める、収まれば緩める** という動的運用。

### context コマンドの使い分け（P4 の具体）

context 圧（要因2）はテキスト化の主要因。Claude Code の context 系コマンドを段階的に使う:

- **`/context`** — 現在の context 使用量の内訳を表示。**テキスト化を疑ったら最初に打つ診断**。
- **`/compact [focus]`** — 履歴を要約圧縮して継続。例: `/compact 直近の作業と diff だけ残す`。生の壊れた出力も要約に置換されるため、要因1(a) と要因2 の両方に効く。CLAUDE.md に compact 方針を書いておくと自動 compact の挙動も制御できる。
- **`/clear`** — 履歴をリセットして fresh context で再開。雪だるま式に悪化して過去履歴が不要なときの **同一セッション内の核オプション**（P1 のセッション移行より軽量）。

> skill 本体（SKILL.md body）は **呼び出し時のみ** context に読み込まれる on-demand 設計で、常時読まれるのは description のみ。使用頻度の低い skill を `disable-model-invocation: true` にして自動起動対象から外せば、description 分の常時読み込みも減らせる（context 削減の補助手段）。

### 速度より確実性が要るとき（仮説・確度低）

- **`/model` 切替** — テキスト化が特定セッション・特定モデルに集中する場合、`/model` で別モデルに切り替える試行は低コスト。公式に tool-call reliability との関係は明記されていない **仮説** だが、効いたら handover に記録する。

## 効かない / 期待しないこと (honesty)

- **hook では捕捉・修復できない** — `PreToolUse` / `PostToolUse` hook は「ツール呼び出しが parse されハーネスに tool-use イベントとして認識された」後に発火する。さらに hook の matcher は **ツール名**（`"Bash"` / `"Edit|Write"` 等）でフィルタする仕組みのため、tool-use イベントとして認識されなかったテキスト化出力には **matcher が適用されず hook が発火しない**。「hook でテキスト化を検知して止める」は hook の設計原理から **原理的に不可能**（claude-code-guide レビューで確認済 2026-06-06）。
- **beatscanner のコード修正では直らない** — ハーネス／モデル層の事象。`backend/` や `frontend/` を触っても無関係（むしろ誤った修正で別バグを生む）。
- **「もう一度同じツールを呼び直す」だけでは再発確率は下がらない** — context が既に汚染されているため。P1（移行）か P3（atomic 化）に切り替える。

---

## 検知 (early warning)

- rendered 出力に **`antml` 等の prefix 文字列・invoke 様タグが地の文として見えたら即停止**。続けると悪化する。
- 「ツールを呼んだのに **結果ブロックが無い**」turn を疑う。
- 同一セッションで **2 回目** が出たら、それは偶発でなくセッション劣化のサイン → P1 を発動。

---

## 再発時の runbook

1. **`/context` で診断 + 即 P3 に切替** — context 圧を確認しつつ 1 メッセージ 1 ツール・preamble 最小・並列をやめる。
2. **`/compact 直近の作業だけ残す` で圧縮を試行** — 生の壊れた出力を要約に追い出す。
3. それでも 1〜2 turn 内に再発 → **`/clear`（軽量）か handover を書いてセッション移行**（P0/P1）。handover には **壊れた断片を貼らない**。「どこまで完了したか」「次の 1 手」だけを atomic に記述。
4. 移行後の新セッションは `/fetch-handover` の圧縮 summary から開始（P4）。full-read しない。
5. 余力があれば **Anthropic へ報告**（§エスカレーション）。最小 repro は別ファイルにメモし、作業 context には貼り戻さない。

---

## エスカレーション / 報告

本事象は **Claude Code ハーネス／モデル層の不具合**。プロジェクト側で根治はできないため、再現性が高いときは Anthropic へ報告する価値がある（claude-code-guide レビューで正規経路を確定済 2026-06-06）。

**報告経路（2 系統）:**

1. **`/feedback`（推奨・一次経路）** — Claude Code セッション内で `/feedback` を打つと Anthropic に直接フィードバックを送れる。
2. **GitHub Issues** — https://github.com/anthropics/claude-code/issues 。再現ステップが明確なときや known issue 確認に使う。メモリ関連なら `/heapdump` 出力を添付。

**添付すべき情報:**

- 症状の記述（テキスト化した turn の概要。**壊れた出力そのものは貼らない** = P2）
- セッション状態（`/context` の使用量・モデル名・継続時間の目安）
- 再現条件（並列ツール数 / preamble 長 など、何をしたら出たか）
- 頻度（同一セッション何回目か、別セッションでも再現するか）

報告時も「最小 repro を作業セッションに貼り戻さない」原則（P2）は維持。

## アンチパターン

- ❌ 壊れた（テキスト化した）出力を「参考に」と次セッション／handover に貼る → priming で再発を持ち越す
- ❌ handover / doc / コメントに **完全なツール呼び出しサンプル** を貼る → 読み戻しで呼び水
- ❌ 1 turn に大量 reasoning + 多数ツールを詰め込む → 崩れ面の増大
- ❌ テキスト化を **hook で直そうとする** → 原理的に発火しない（時間の無駄）
- ❌ 「コードのバグだ」と誤認して beatscanner の backend/frontend を触る → 無関係・別バグの温床
- ❌ 症状が出ているのに同じ重い turn を粘り続ける → 雪だるま式悪化。早めに移行
