## 🟡 重要知見 (永続化済、 本 session 教訓)

### article_pipeline は narration layer (LLM 持つ) だが数値計算指示は BLOCK
- aggregator/ (# @no-llm) と異なり article_pipeline は LLM SDK を import OK
- ただし system prompt 内の **「計算してください」 imperative** は BAD-3 数値捏造を誘発
- → pre-commit hook Check 4 で imperative 表現を BLOCK、 negation 文脈 (「行わない」「そのまま引用」) は除外
- 数値は呼出側 (scheduler.py) が事前計算 + raw_sources に詰めて渡す、 LLM はそのまま引用

### Researcher URL whitelist = Hallucination Guard 第 4 層の本丸
- LLM が citation URL を捏造する pattern (BAD-3 直撃 zone) を post-process で確実に弾く
- raw_sources の `url` field を set 化、 LLM 出力 citation の `source_url` がこの set に無ければ破棄
- confidence < 0.7 の Citation を含む SourceFact も破棄 (低品質 source は記事化させない)

### Writer [N] citation enforcement
- body_md 内の数値・固有名詞 sentence の直後に `[N]` (N は source_facts の 1-起点 index) を埋め込ませる
- post-process で `[N]` が 1 つも無い draft は ValueError 即破棄 (citation 欠落は記事として publish 不可)
- json_indexes と body_indexes の整合 + range check (1 ≤ N ≤ n_sources) で 3 重検証

### Supabase 連携は silent-fail で pipeline を死なせない
- ENV (`SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY`) 未設定時は upsert スキップ + log
- migration 未実行時の INSERT 例外も catch + log、 pipeline response は返す
- → P2 deploy 後でも frontend で route 公開可能、 storage は段階的に活性化

### GitHub Actions cron は HTTP trigger pattern
- 旧 design (Python script + articles/ commit) は scale しない (Researcher / Writer / FactChecker 全部 Actions runner で実行 = Anthropic SDK install + API key secret + 15 min timeout)
- 新 design = 既存 `morning_health.yml` pattern 流用、 curl で endpoint hit するだけ
- Railway 側で pipeline 実行 + Supabase 保存、 Actions runner は trigger + response 確認のみ
