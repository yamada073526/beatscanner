## 1. v88 で確立した永続教訓

### A. Evaluator subagent 内蔵 3 体合議 pattern (v88 新発見、 memory 起票推奨)

memory 名候補: `feedback_evaluator_inline_3_panel.md` または `feedback_multi_review_3_panel_workflow.md` 拡張

Sprint B Evaluator が L4 simplified を **3 体並列起動 (UI/UX + 設計 + 開発) で内蔵実行** し、 verdict を集約。 これにより:
- sprint 単位の Evaluator pass で 3 体合議カバレッジ確保
- Phase 完了 gate での別途 3 体合議起動を **省略可能**
- timeflow: Evaluator 30-40 min vs 3 体合議 30-40 min の **重複排除**

適用条件: 3 軸 (LLM 出力 / Trust Cliff / 新 backend) で 0-1 active = 3 体合議で十分判定の sprint に限る。 6 体合議が必要な sprint (LLM prompt 変更 / Phase gate 重大判断) では Evaluator 内蔵では不足。

### B. backend `_periods_built` の 3 ルート構造 (v88 で全 mapping 確認、 memory 起票推奨)

memory 名候補: `backend_periods_built_routes.md` または既存 `tech_stack.md` 拡張

`backend/app/main.py` の `_periods_built` array は 3 ルートで構築される:
1. **FMP "instant" route** (line ~8157): `_shr = _inc.get("weightedAverageShsOutDil") or _inc.get("weightedAverageShsOut")`
2. **yfinance "instant" fallback** (line ~8206): `shares = _g(inc, col, 'Diluted Average Shares', 'Basic Average Shares')`
3. **FMP visualize route** (line ~8492): `_shr = _inc.get("weightedAverageShsOutDil") or _inc.get("weightedAverageShsOut")`

新 field を `_periods_built` に追加する場合、 **3 ルート全てに追加必須**。 1-2 ルートのみだと特定 endpoint / fallback 経路で NaN / undefined になり frontend 描画失敗 (v88 SPS bar 透明事故の root cause)。

### C. SPEC vs 実装の minor 差分は実装側が優れる場合あり (v88 で `>999%` → `>-999%` 発見)

SPEC で「1000% 超は `>999%` で打ち切り」 と記載したが、 実装で `>+999%` / `>-999%` の符号付き表記に拡張。 これは負方向の extreme 値で符号が失われる SPEC bug を実装で fix した形。 verdict: SPEC を事後更新推奨。

教訓: 「SPEC は完璧でない、 実装中の発見で SPEC 改善する pattern を受容する」。 PGE 3 体ループでは Generator が SPEC を文字通り実装するのでなく、 **より良い解を見つけたら deviate して OK** という運用方針。
