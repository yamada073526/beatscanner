### 4. published_at NULL hotfix (Phase 14-15、 v123 追記)

**真因**: Phase 14 (24h 健全性確認) で Supabase の `published_at IS NULL AND status='published'` 状態の article 2 件を発見:
- `googl-202605241934` (5/24 生成)
- `nvda-202605251039` (5/25 生成)

これは v121 末期に user が service_role で status='draft' → 'published' に手動更新した時に `published_at` column を update し忘れた残骸。 v122 で自動 publish された article (5/26 22:30 published_at) は正常。

**症状**: `ArticleHero.formatPublishedAt(null)` が空文字列を返し、 `<time>` tag が non-render = ユーザー視点で「日付不明」 状態。 軽微だが brand 信頼に微影響。

**hotfix (commit 0b855d6)**: `ArticlePage.jsx` の setArticle() 全 4 path に `withPublishedAtFallback(data)` を適用:
- SSG inject path
- article-data.json fetch path
- Supabase fallback path (v122 既存)
- network error fallback path

`published_at` が null/undefined なら `generated_at` で fallback、 両方 null なら null 維持。

**Supabase 直接 backfill は保留**: auto mode classifier が privileged write を block (適切な safety 挙動)。 user が Supabase dashboard で以下 SQL 実行で完全解決:
```sql
UPDATE articles SET published_at = generated_at
WHERE published_at IS NULL AND status = 'published';
```
ただし frontend hotfix で同等 UX が出るため緊急性なし。

**Hotfix 2 段**:
- Step 1 (commit 0bc8c15): max_tokens 512 → 1024 で META 8 facts (行 35) カバー
- Step 2 (commit 4534871): MSFT 11 facts (行 96 truncate) でまだ不足判明 → 1024 → **2048**

**Why 2048 で十分** (mathematical):
- META 8 facts → 行 35 (~512 tok 超過僅か)
- MSFT 11 facts → 行 96 (~1024 tok 超過)
- 比例則: 1 fact ≈ 90 tok → 20 facts でも 1800 tok 程度 → 2048 で覆う
- cost +$0.005/article × 1 article/day × 30 day = 月 150-500 円 negligible

**5/28 朝 cron での natural verify**:
- DOY 148 (5/28) → ROTATION[148%5=3] = **GOOGL** 自動生成 UTC 19:05 = JST 04:05
- 8-12 facts 想定で 2048 で fact_check passed 期待 → status='published' 自動化
- article-notify cron UTC 22:00 (= JST 07:00) で email 配信 → user 朝起床時に通知

**メモリ anchor**: [feedback-fact-checker-token-truncate](feedback_fact_checker_token_truncate.md) で永続化
