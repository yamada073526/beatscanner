## 🟢 5 機能着地内容詳細

### 1. article auto-publish (storage.py + router.py)

**真因 (v121 dogfood 発覚)**: cron で生成された article は backend 200 OK で完走するも `status='draft'` で滞留 → frontend は `status='published'` でしか fetch しない → user 視点で「記事が無い」 状態。 user が service_role で手動更新するまで blackout。

**実装**:
- `backend/app/article_pipeline/storage.py:148-167`
- `final_status='passed'` (fact_check.passed=True、 Hallucination Guard 4 重防御 完全通過) → **`status='published'` + `published_at=generated_at`** で自動 publish
- `final_status='regenerate_failed'` (retry 後も mismatch 残存) → `status='draft'` で人間 review 必須維持
- `verdict_sign` は block しない設計 (balanced_view_needed=True で警告のみ) のため fact_check passed だけで安全

**Hallucination Guard 安全性**:
- 既存 4 重防御 (pre-commit hook / NEGATIVE_EXAMPLES BAD 1-6 / frontend sanitize / sources schema) は不変
- storage.py の status 制御のみ追加、 pipeline 自体 unchanged
- v121 sample 4 件で文体憲法 v2 品質確認済 (deep_dive nvda-202605262204 が「完璧反映」 verdict)

## 🟡 重要知見 (永続化済、 memory anchor 候補)

### cup-scan は Railway native cron で動いていた (v121 で path 誤記)
v121 で「.github/workflows/rs_scan.yml」 と書いたが実際は **Railway native cron** (`railway.toml` の `[[crons]]`) で動いていた。 GitHub Actions ベースの cron (generate_article.yml / generate_daily_digest.yml) と Railway cron (cup-scan / cup-notify / movers warmup 等) の 2 系統が混在。 新規 cron 追加時は cup-scan pattern (Railway native) を流用する方が低コスト。

### article auto-publish の Hallucination Guard 安全境界
`final_status='passed'` は fact_check.passed=True を含意 (scheduler.py:177 で確定)。 これは pipeline 内で:
- writer → fact_checker (LLM 2 段)
- mismatches なし or 再生成で解消
- pre-commit hook + NEGATIVE_EXAMPLES + sanitize + sources schema 全て enforced

の 4 重防御をすべて通過した状態。 status='published' 自動化は safe。

ただし **verdict_sign は block しない設計** (line 173-175、 balanced_view_needed=True で警告のみ) のため、 「強気 article + judgment_pass=False」 矛盾でも publish される。 これは Writer 側で両論併記が生成される設計のため意図通り (handover v82 verdict)。

### Railway redeploy が遅れる場合の SSG fallback 設計
build-articles.mjs (vite build 後 post-process) は deploy 時のみ SSG する。 cron で publish された article は **次 deploy まで dist/articles/<slug>/ 不在 = 404**。 解決策 2 系統:
- A. Railway deploy webhook を backend cron で trigger (Railway service settings で webhook URL 生成、 env var に set、 user 操作必要)
- B. **frontend で Supabase fallback fetch** (v122 採択、 implementation cost 低、 SEO は失うが UX 維持)

将来 SEO / OGP 重視するなら A も並走可能。

### article-notify は cup-notify と同じ Resend pattern + 同じ dedup table 流用
新 schema migration 不要。 `notification_dispatch_log` の既存 column を `pattern_type='article'`, `ticker=slug` の semantic で再利用。 1 user 1 slug 1 channel = 1 回まで送信。
