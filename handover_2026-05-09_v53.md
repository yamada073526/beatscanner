### IO バグの本質
旧実装 (v52): `Math.abs(0.5 - intersectionRatio)` で「画面中央」を判定 → 大型カードは「画面に少し入った状態」と「画面外に出かけた状態」が両方 ratio≈0.5 になり、過去のセクションに発光が戻る。
新実装 (v53): `getBoundingClientRect()` で要素中央 Y と viewport 中央 Y の距離を計算、最小距離の **leaf** カード 1 枚だけを active 化。`r.bottom <= 0 || r.top >= vh` で完全画面外を除外 (dead zone)。

### 発光演出の方針 (3 体合意)
- **タブ・セクション横断で統一** (Apple HIG / Linear / Stripe Dashboard 流) — 使い分けはユーザー学習コスト増の原因
- **トリガー責務分離**: arrival (scroll IO) = 弱、hover = 強、focus-visible = a11y、state-change = data tick flash
- **leaf 優先**: 子発光がある時は親発光を抑制 (`:has()` selector + `contain: layout paint`)
- **PortfolioHistoryChart は子として独立発光** (親 `.bs-panel` は子発光時に静止)

## 3. 🟡 重要知見 (v53 セッション固有)

### 3-1. IntersectionObserver は「集合管理」専用、「中央判定」には使わない
`intersectionRatio` は (target ∩ root) / target なので、長尺カードでは **画面に少し入った状態と画面外に出かけた状態が両方 0.5** になる。「画面中央 1 枚」の判定には不向き。
→ IO は `threshold: 0` で「視認可能集合」を維持するだけ、active 判定は `getBoundingClientRect()` の center 距離で scroll/resize 駆動の rAF tick で計算。

### 3-2. `:has()` selector で親子発光競合を解決
```css
.bs-panel:has(.surface-card.is-arriving) { box-shadow: none !important; }
```
Safari 15.4+ / Chrome 105+ / Firefox 121+ で利用可。beatscanner の対象環境では問題なし。

### 3-3. `contain: layout paint` で paint isolation
`.surface-card { contain: layout paint; }` で子発光が親に染み出さず、GPU compositor を独立 layer 化。CLS=0 維持にも貢献。

### 3-4. Apple Pay 流の sheen sweep の cubic-bezier
分析 CTA で採用: `transition: transform 600ms cubic-bezier(0.16, 1, 0.3, 1)` で `translateX(-120%) → translateX(120%)` を hover 時にトリガー。Stripe "Continue" ボタンと同じ手法。

### 3-5. `data-urgent="true"` で amber pulse データ駆動
決算 7 日前以内 (`daysToEarnings <= 7`) で `data-urgent="true"` を JSX で付与 → CSS `[data-urgent="true"]` で amber pulse animation。Bloomberg Terminal 級の「データ駆動演出」を非破壊で実装可能。
