## 5. 重要バグ修正 (3 件)

### 5.1 「最終更新 20556 日前」表示異常 (epoch 秒誤解釈)
**原因**: `/api/movers` の `updated_at` は **Unix エポック秒** (例: 1777880534) なのに、`new Date()` で **ミリ秒** として解釈 → 1970-01-01 (UNIX エポック) → 約 56 年差 = 20556 日。

**修正**: `formatRelativeTime` に input 型判定追加
```javascript
if (typeof input === 'number') {
  const ms = input < 1e12 ? input * 1000 : input;  // 10桁=秒, 13桁=ms
  then = new Date(ms);
}
```
時計ずれ保険 (`if (diffMin < 0) return 'たった今';`) も追加。

### 5.2 「登録不要で試せる」虚偽訴求 (Trust Cliff 再発)
**問題**: LP の「今日の注目」「今週の注目決算」「サンプル分析」「あなたが見た銘柄」をクリックすると、`runAnalyze` が `if (!hasFmpKey()) setShowApiKeyModal(true)` で停止 → 「APIキー必要」モーダルが出て約束を裏切り。

**修正**: App.jsx に `handleLPTickerClick` ハンドラ新設
- LP からのクリック専用 (今日の注目 / 今週の決算 / サンプル分析 / あなたが見た銘柄)
- `hasFmpKey()` あり → 通常 `analyze` (制限なし)
- `hasFmpKey()` なし → `demoAnalyze` (3銘柄/日制限・登録不要)
- localStorage に bs_analyzed を記録
- 429/limit エラー時は「本日のお試し回数を超えました。Googleログインで無制限に。」表示

LP の `onTickerClick` prop 経由で全 4 セクションから呼び出し。

### 5.3 デモモードの銘柄制限が LP 訴求と矛盾
**問題**: LP では「✓ 3銘柄/日まで無料分析」と表記しているのに、バックエンド `/api/demo/analyze` は AAPL/MSFT/NVDA の 3 銘柄に固定 → 「今日の注目」(MSTR/ORCL/GME 等) クリックで「3銘柄のみ利用できます」エラー。

**修正**: バックエンド `/api/demo/analyze` のホワイトリストチェックを撤廃
- 任意銘柄を分析可能に (悪用防止は IP ベース rate limit 3 req/IP/day で担保)
- エラーメッセージも統一: 「本日のお試し回数 (3銘柄) を超えました」

## 6. パフォーマンス改善 (3 段階)

### 6.1 第1段階: プリフェッチ拡張 + result キャッシュ
**問題**: 各タブの読込が 5-10 秒。原因は直接クリック時 prefetch が走らず、各 panel が mount 後に個別 fetch する直列待ち時間の累積。

**改善 (3 つ)**:
1. **prefetchAll を 3 → 7 endpoints に拡張**
   - 旧: guidance/basic + chart/summary + insights
   - 新: 上記 + news + ir-links + price-history + analyst
2. **runAnalyze と handleLPTickerClick の冒頭で prefetch() を即実行**
   - hover 時のみだったプリフェッチを、直接クリック時にも適用
   - analyze と panel データ fetch が並列化
3. **result キャッシュ (10 分 TTL) を `useRef` Map で実装**
   - 同銘柄の再訪が瞬時 (0 秒)
   - ウォッチリスト回遊時の往復が高速化

**効果**:
- 初回分析: 10s → 7s
- 再分析 (10 分以内): 7s → **0s 瞬時**
- ウォッチリスト回遊: panel が HTTP cache HIT → 1-2s

### 6.2 第2段階: movers warmup + 並列度向上
**問題**: 「今日の注目」が cold cache 時 10-15 秒かかる。

**改善 (2 つ)**:
1. **Railway cron で `/api/movers` を 15 分毎に warmup**
   - キャッシュ TTL 20 分 → 15 分毎で常時 warm
   - ユーザーは常に 300-500ms 応答
2. **/api/movers の batch 処理を高速化**
   - batch_size: 4 → 8
   - batch 間 sleep: 1.0s → 0.3s
   - cold 時応答: 14s → ~6.6s (約 2x 高速化)

### 6.3 第3段階: 初回ロード 10s → 5s 目標 (3 つの施策)
**問題**: ユーザーから「最初に開いた時 10 秒」報告。

**改善**:

**A. Backend keepalive cron**
```toml
[[crons]]
schedule = "*/10 * * * *"
command = "curl -s .../health > /dev/null"
```
Railway インスタンス休眠を防止 → cold start 5-10s 排除。

**B. React.lazy で重いコンポーネント遅延読み込み**
App.jsx で 6 コンポーネント:
- LandingPage (1403行) — ログイン済みは不要
- DetailReport (960行) — 決算タブクリック時のみ
- ChartTab (669行) — チャートタブのみ
- CalendarPanel (328行) — カレンダー表示時のみ
- ScreenerPanel (128行) — スクリーナー表示時のみ
- CustomScreenerPanel (215行) — カスタムスクリーナー表示時のみ

HomeTab.jsx で 2 コンポーネント:
- DiagramCard (2027行) — AI 図解表示時のみ
- ChartTab (669行) — ウォッチリストチャート展開時のみ

各箇所 `<Suspense fallback="読込中...">` で囲む。

**C. Vite manualChunks で vendor split**
```javascript
manualChunks: {
  'react-vendor': ['react', 'react-dom'],   // 137KB
  'supabase': ['@supabase/supabase-js'],    // 201KB
}
```

**バンドル分割結果 (本番実測)**:
| Chunk | 初回 DL | 2 回目 (cache) |
|---|---|---|
| index | 706KB | 706KB |
| react-vendor | 137KB | **0** |
| supabase | 201KB | **0** |
| lightweight-charts | 197KB | 0 (チャート時のみ) |
| DiagramCard | 44KB | 0 (AI 図解時のみ) |
| DetailReport | 29KB | 0 (決算タブのみ) |
| LandingPage | 23KB | 0 (未ログイン時のみ) |
| ChartTab | 13KB | 0 (チャートタブのみ) |
| CalendarPanel | 7.6KB | 0 |

**期待効果**:
- 初回 (cold): 10s → **3-5s**
- 2 回目以降 (cached): **1-2s**
- タブ遷移: lazy chunk 5-30KB の追加 DL のみ

## 10. v39 で確立した永続ルール候補 (CLAUDE.md 昇格検討)

### 10-1. Trust Cliff (信頼の崖) は最重要バグ
- 「登録不要」と書きながら登録要求モーダルを出す → CVR 30-40% 落ちる可能性
- 「3銘柄/日まで無料」と書きながら 3 銘柄ホワイトリスト → 即離脱
- LP の訴求と実装が**完全に一致**しているか毎回確認すること

### 10-2. 投資業界の色ルール
- 上昇 = **緑 (`#34ef81`)**
- 下落 = **赤 (`#f87171`)**
- 中立 = シアン or グレー
- 警告/緊急 = **amber (`#f59e0b`)** (例: 決算が X 日後)
- シアンは「ブランド色」として使うが「上昇」を意味しない

### 10-3. 動的データには「最終更新 X 分前」を併記
- 「データが動いている感」がリテンションに直結
- epoch 秒/ms の自動判定パターン: `input < 1e12 ? input * 1000 : input`

### 10-4. ログイン後はキャッチコピーを出さない
- Apple/Notion/Linear/Stripe/Figma 全社が採用
- ログイン後は「使うモード」: 検索バー・データ・操作要素を最優先
- LP の Hero は未ログイン時のみ価値がある

### 10-5. プリフェッチ運用
- 重い API は **必ず prefetchAll に含める** (現状 7 endpoints)
- ユーザー操作 (hover / 銘柄クリック / ウォッチリスト追加) の時点で fire-and-forget 起動
- 直接アクション (form submit / LP click) でも runAnalyze 等の冒頭で必ず prefetch 呼ぶ

### 10-6. result キャッシュ (10 分 TTL)
- `useRef(new Map())` で App.jsx 内に保持
- 同銘柄の再訪を 0 秒化
- F5 (リロード) で消える設計 (memory cache)

### 10-7. 無料お試しは「IP ベース rate limit」だけで十分
- ホワイトリスト方式 (AAPL/MSFT/NVDA 限定) は LP 訴求と矛盾しやすい
- バックエンドの demo endpoint は **任意銘柄 + 3 req/IP/day** が標準
- 内部処理が高コスト API (Claude 等) を含む場合は別途考慮

### 10-8. Backend cold start 防止
- Railway 等のサーバレス系は無アクセス時にインスタンス休眠 (5-10s 起動遅延)
- 10 分毎の `/health` cron で常時 warm 維持
- 重い API のキャッシュは別 cron で warmup

### 10-9. コード分割 (React.lazy) の判断基準
- 該当タブ/モーダルを訪問しない可能性が 50%+ → lazy 候補
- 行数 200+ → lazy で初期バンドル軽量化
- DiagramCard / DetailReport / LandingPage は最強の lazy 候補

### 10-10. Vite manualChunks の標準
- `react-vendor`: react + react-dom
- `supabase`: @supabase/supabase-js
- `lightweight-charts`: 既に独立 chunk (Vite 自動)
- 他のライブラリは初期バンドルに含めて OK
