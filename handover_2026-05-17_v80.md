## 3. 触ると危険な箇所 (v80 新規 + 継承)

### v80 新規確定原則

**Cup-Handle Phase 2 nightly scan の rate limit (handover v80、 multi-review SRE verdict)**:
- yfinance primary、 chunk_size=10 / sleep 1s で 200 銘柄 ~30 秒
- Railway IP block 時の fallback (FMP primary 化) は Phase 2.6 で実装、 現状は失敗銘柄 skip + log のみ
- `BACKTEST_PHASE2_UNIVERSE_TOP200` (main.py:2885) を勝手に増やすと scan 時間が線形増加、 200 → 1000 で 2-3 分かかる

**ファンダ 5 PASS × transition AND default (handover v80、 multi-review SaaS PM verdict)**:
- cup-notify は default で `_is_ticker_funda_pass()` を呼び、 earnings_evaluation で all_passed=true (直近 95 日) の銘柄のみ通す
- debug 時のみ `{"skip_funda_filter": true}` で全 transition 通す (本番 cron では絶対付けない)
- これを外すと月 100+ 通知 → 「狼少年化」 → Premium churn 直撃

**Resend mail 送信頻度ガード (handover v80、 multi-review Trust Cliff verdict)**:
- 同 (user, ticker, transition_type) で **7 日 dedup** (notification_dispatch_log で sent status 確認)
- 1 user 1 日 1 通 digest 集約 (per_user_queue で aggregation)
- 解約即停止: `subscriptions.tier='premium' AND status='active'` を送信時 fetch (cache 使用禁止)
- これらを external dependency に任せず backend 内に保持しているのは「Premium churn = 売上直撃」 のため

**backend payload mask (handover v80、 multi-review Security verdict)**:
- `/api/scanner/cup-handle` で `is_premium=false` user には pivot/state 詳細を **backend response 段階で除外** (`_mask_signal_for_free()`)
- frontend CSS blur だけだと DOM/DevTools で剥がれる → Premium 価値情報 leak
- 新規 scanner endpoint を作る場合は必ず同 pattern (backend mask + visible_count + total_count) を踏襲

**`pattern_signals` RLS service_role only (handover v80、 multi-review Security verdict)**:
- authenticated にも read 許可しない (Premium gate を backend API SSOT に集約)
- 例外的に notification_dispatch_log は authenticated × SELECT 許可 (RLS で `auth.uid() = user_id` filter、 将来 in-app inbox 用)

### v74-v79 から継続 (handover v79 §3 参照、 削除可)

- Recharts overlay 4 層防御 (handover v75 真っ白事故 SSOT)
- `<Line data={別 array}>` 禁止 (v76 教訓)
