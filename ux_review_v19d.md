# UXレビュー beatscanner v19d

レビュー日時: 2026-04-29
対象コンポーネント: `DiagramCard.jsx`（v19c の `DetailReport.jsx` VizPanel から置き換えられた DOM ベース実装）

---

## 🔴 重大（すぐ直すべき）

### 1. 売上高チャート — 営業利益率ラベルとYoYラベルの垂直重複

`BarChartWithMargin` の垂直レイアウトを実測すると次の問題がある。

営業利益率ドットラベルは `y = opToY(d.value) - 6` に配置される。`OP_AREA_BTM = 62` が実質的なスパークラインの下限なので、ラベルは最悪 `y ≒ 56` に来る。  
一方、YoYラベルは `y = by - 10` だが `by` の最小値は `BAR_AREA_TOP = 72`（最大棒高のとき）なので `y ≒ 62` となる。  
「OP_AREA_BTM（62）」と「YoY最小y（62）」が一致しており、**棒が最大高（バー3本の値が近接している場合）のとき、スパークラインの一番下のドットラベルとYoYラベルが同一ライン上に乗る**。

スクリーンショットで確認できる「41.3% / 48.4% / 48.3%」の各ラベルと「+15.7% / +14.9%」のYoYラベルの縦方向の詰まりはこの計算上の余裕ゼロに起因する。

**改善案**:  
`OP_AREA_BTM` を `62 → 56` に、`BAR_AREA_TOP` を `72 → 78` に変更し、スパークラインバンドとYoY帯の間に6px以上のクリアランスを設ける。コードの `// touches band floor, no overlap` コメントは現状では誤り（exactly touches = 重なりうる）なので修正すること。

```
// DiagramCard.jsx BarChartWithMargin
const OP_AREA_BTM    = 56;   // 62 → 56 (6px クリアランス確保)
const BAR_AREA_TOP   = 78;   // 72 → 78
const SVG_H          = AXIS_Y + 46;  // AXIS_Y = 78+110=188, SVG_H=234 (+6px)
```

### 2. 売上高チャート — 「営業利益率▶」凡例と最初のドットラベルの水平衝突

凡例テキスト `営業利益率▶` は `x = SVG_W - RIGHT_PAD, y = MARGIN_LABEL_Y(28)` に `textAnchor="end"` で描画される。  
一方、3番目（最右）のドットラベルは `x = cxArr[2]` かつ `y = opToY(d.value) - 6`。cxArr[2] は `SVG_W - RIGHT_PAD - BAR_W/2 = SVG_W - 32` 程度であり、バー幅44px・テキスト幅（"48.3%"約28px）を考慮すると凡例ラベルと**横方向で重なる可能性がある**。スクリーンショットのチャート右上で"営業利益率▶"と"48.3%"が密接しているのはこのため。

**改善案**:  
凡例を右上固定から**左上固定**（`x = LEFT_PAD, textAnchor="start"`）に移動する。凡例はチャートの左上に置くほうが、欧米・日本いずれのチャート慣習にも合致し、ドットラベルとの衝突を構造的に解消できる。

---

### 3. EPS/CFPS/営業CFチャート（`BarChartPanel`）— バー値ラベルが初回バーにのみ表示

`BarChartPanel` は `BarChartWithMargin` とほぼ同じ構造だが、`BarChartPanel` コンポーネントに **`BAR_TOP` 変数が未定義のまま `{maxV}` のY軸ラベルで参照されている**（L82: `y={BAR_TOP + 8}`）。

`BAR_TOP` は `BarChartWithMargin` のみに定義されており `BarChartPanel` には存在しない。これは実行時エラーにはならない（undefined + 8 = NaN → SVG属性として無視される）が、Y軸最大値ラベルが描画されないため**スケール感が失われる**。投資家がバーの絶対値を読み取れず、成長率のみで判断せざるを得ない状態になっている。

**改善案**:  
`BarChartPanel` の `BAR_TOP` を `BAR_AREA_TOP`（=54）に統一するか、変数名を揃える。

```js
// BarChartPanel 内 (L82付近)
<text x={LEFT_PAD - 4} y={BAR_AREA_TOP + 8} ...>   // BAR_TOP → BAR_AREA_TOP
```

## 🟡 中程度（次スプリント）

### 4. チャートグリッド — モバイル幅でのカード内パディングの狭さ

2×2グリッドの各チャートカードは `padding: '8px 8px 4px'`。モバイル（375px幅）では1カードあたり約`(375 - 32 - 8) / 2 = 167px`の幅になる。  
`BarChartWithMargin` の `SVG_W = LEFT_PAD(40) + 3*BAR_W(132) + 2*SPACING(40) + RIGHT_PAD(10) = 222px` に対し、カード内利用可能幅は167pxしかなく、**SVGがカード境界をはみ出す**。`overflow: 'visible'` のため見た目は壊れないが、隣のカードとSVGが視覚的に重なる可能性がある。

**改善案**:  
グリッドのギャップを `8px → 6px` に縮小し、外側パディングを `16px → 12px` に縮小する。または375px以下では `gridTemplateColumns: '1fr'` の1カラム表示に切り替える（チャートが縦に積み重なるが各チャートは十分な幅を確保できる）。

---

### 5. バリュエーションカード — 判定バッジの基準が非明示

PER/PBR/PSRカード（L407–443）の `judgeColor` は `割安 / 割高` の二値で色分けされているが、「何倍以上が割高か」の基準値がUI上に一切表示されない。ユーザーが「PER 32.5xはなぜ割高なのか」を確認する手段がない。v19cで既に指摘済みだが `DiagramCard.jsx` への移行後も未解決。

**改善案**:  
各カードに `title` 属性（ホバーツールチップ）として基準値を付与する（例: `title="業種中央値 PER ≈ 28x"`）。または `ConditionCard` 同様の `？` ボタン + InfoModal を追加する。工数が大きければ `title` 属性だけでも即効性がある。

---

### 6. ビジネスモデルフロー — スクロールヒントが常時表示

`scroll-hint-mobile` クラスを持つヒントテキスト「← スクロールして全体を確認」は CSS クラスで出し分けているが、実際の描画は `display: 'flex'` がインラインスタイルで設定されていないため**常時表示**されている状態になりえる（クラス側のメディアクエリ定義に依存）。デスクトップでも4ボックス程度ならスクロール不要なのにヒントが出ると、ユーザーに誤ったメンタルモデルを与える。

**改善案**:  
`scroll-hint-mobile` をデフォルト非表示（`display: none`）にし、モバイルのみ表示するメディアクエリを `index.css` に明示するか、`isMobile` 変数を使って条件付きレンダリングにする。

```jsx
{isMobile && (
  <div style={{ fontSize: '11px', color: 'var(--text-muted)', marginBottom: '6px' }}>
    ← スクロールして全体を確認
  </div>
)}
```

---

### 7. 強み・リスク列 — モバイル判定がウィンドウリサイズに非反応

`isMobile` の判定（L320）は `window.innerWidth < 480` をレンダリング時に一度だけ評価する。Reactのstateではないため、ユーザーがブラウザ幅を変更したりデバイスを回転させても**レイアウトが再計算されない**。特にタブレット横向き（768px）→縦向き（1024px）のような回転では崩れが残る。

**改善案**:  
`isMobile` を `useState` + `useEffect` の `resize` イベントリスナーか、CSS `@media` クエリのみで制御するリファクタリングを行う。CSS Grid の `@media (max-width: 479px) { grid-template-columns: 1fr; }` で対応するほうが宣言的で堅牢。

---

### 8. 全体情報密度 — セクション間のスクロール距離が長い

ページ構成：ヘッドライン → バリュエーション → ビジネスモデルフロー → 2×2チャートグリッド → 強み/リスク → 投資家Q → Bull/Bear。これをすべて縦に並べると、モバイル375px幅で推定スクロール距離は2000〜2500pxに達する。

投資家ユーザーの主要ユースケース（「PASSかFAILか確認 → 数字を確認 → 売る/買うの判断」）では、PASシグナルを確認した後、バリュエーションとチャートだけを素早く見たい。強み/リスク・Bull/Bearは補助情報だが、チャートと同じ縦スタックに置かれるため「結論に到達するまでのスクロール量」が多い。

**改善案**:  
強み/リスク・Bull/Bearセクションをデフォルト折りたたみ（`<details>` または手動アコーディオン）にし、「詳細を見る ▼」トグルで展開する。ヘッドライン〜チャートグリッドの4セクションだけなら1000px以内に収まる。

## 🟢 良い点（維持すべき）

- **`BarChartWithMargin` の垂直レイアウト設計思想**: 営業利益率スパークラインをバー上部に専用バンド（y=38〜62）として確保し、バー本体と分離しようとしたアーキテクチャは正しい。微細なオフセット調整だけで完成する。
- **バリュエーションカードの数値サイズ**: PER/PBR/PSRを `fontSize: 18px, fontWeight: 700` で表示し判定を色バッジ化した構成は、スキャン速度が高くコンパクト。3〜4カードのflex wrapsレイアウトも幅変動に追従しやすい。
- **ビジネスモデルフローの水平スクロール実装**: `overflowX: auto` + `WebkitOverflowScrolling: touch` + `width: fit-content` の組み合わせはiOS慣性スクロールに対応した正しい実装。`flex: 0 0 auto` でボックスが縮まない点も適切。
- **ヘッドラインカードのレスポンシブタイポグラフィ**: `fontSize: clamp(18px, 5vw, 28px)` は狭い画面でも広い画面でも読みやすいサイズを確保する良いアプローチ。
- **BEAT/MISSラベルのカラーコード**: `▲BEAT` を `#10B981`（緑）、`▼MISS` を `#ef4444`（赤）で統一し、アプリ全体のカラートークンと一致している。
- **強み・リスクのブレット色分け**: 強みを `#22c55e` のドット、リスクを `#ef4444` のドットで視覚的に区別しており、テキストを読まずとも色だけでセクション帰属が判断できる。
