### Vision-eval scorer の安定性改善

v97 で snap-vision-eval.mjs を新規実装したが、 3 cycle PDCA で以下の bias が判明:

1. **frame count diff で scoring 基準ばらつき**: ticker 切替で「会社概要 accordion」 を探せない場合 5 frame、 探せた場合 8 frame。 5 frame だと motion 軸の判断材料が少なく厳しめ scoring (NVDA で常時 -5 pt 低い)。
2. **session 内 drift**: baseline 9 sample のうち AAPL run1 (76.6) / MSFT run1 (78.6) が突出、 残 7 sample は 65-71 (mean 67-68)。 「初回甘め、 2 回目以降 厳しめ」 pattern が見える。
3. **3 run mean variance ±4 残る**: feedback_vision_api_noise.md SSOT は ±2 まで縮まる想定だが実測 ±4。

**改修方針 (次セッション)**:
- a. frame count を必ず 8 で揃える (accordion open 失敗時の fallback section を追加)
- b. anchor 画像 base scoring: 「これが 50 点」「これが 80 点」 のリファレンス画像を rubric に同梱
- c. session 間で fresh chromium context 強制
- 推定工数: 60-90 分

これが解決すれば 100 点 PDCA は dependable な measurement で回せる。

### user 指摘 3 件 fix の詳細

1. **#1 CLS (画面ガクつき) 解消**
   - 真因: JudgmentDetail.jsx の `{guidance && (<GuidanceCard ...>)}` conditional render が data 取得まで mount せず、 guidance 後追い load で縦幅 0 → 数百 px ジャンプ → 下にある ProfileCard click target が押し下げられる
   - 修正: 常時 mount + `isLoading={!guidance && detail?.isLoading !== false}` で 3 state (skeleton / no-data / ok)
   - GuidanceSkeleton も raw `bg-slate-200` (Tailwind/design-system 違反) → token + gold shimmer + minHeight 360px

2. **#2 会社概要 shimmer のっぺり改善**
   - 改善 3 点:
     - 実 output shape 化 (lead + 4 sections × 3px gold accent + 2 lines)
     - 進捗 step text 2 秒おき切替 (「FMP 企業データを取得中」 → 「AI 要約生成中」 → 「もう少しで完了」)
     - shimmer wave 1.5s → 1.2s で 25% 高速化

3. **#3 30 秒 loading 真因 fix**
   - 各 endpoint latency 計測 (本番 curl + time):
     - 🔴 `/api/insights/MSFT`: cold **19.00s** (Claude LLM call、 warm 0.44s)
     - 🟡 `/api/guidance/MSFT`: cold 7.16s
     - その他 /api/analyze 0.78s / /api/profile-summary 0.72s / /api/news 1.45s
   - 「30 秒近く」 の正体 = cold start 5-10s + insights 19s
   - 修正: `refresh_insights_batch` cron を「watchlist 20 件」 → 「watchlist + S&P 500 top 30 = 最大 50 件」 に拡張、 主要 ticker は永続 supabase cache 保持で 19s → 0.5s
   - cost: 50 銘柄 × Claude Haiku $0.005-0.01 = $0.25-0.5/day = ¥40-80/day (¥1200-2400/月)

## 4. 永続教訓 (v97 で確立、 memory anchor 候補)

### A. Vision-eval scorer の frame count 依存 bias [新規 memory: feedback_vision_eval_frame_count.md]

scorer で ticker 切替時に accordion が open 既定の場合 frame 5、 closed default の場合 frame 8 → motion 軸基準が変わる。 NVDA 常時 -5 pt 低い真因 = frame 数不足。

**対策**:
- accordion open は必ず成功させる (fallback selector: 「会社概要」 → 「最新ニュース」 → 任意 collapsed section)
- 失敗時は frame 数を 8 に padding (静止 frame 重複でも基準統一)
- session 間で fresh chromium context (cookies / cache reset)

### B. FMP API path Legacy 認定の silent fail [新規 memory: feedback_fmp_api_legacy.md]

FMP は 2025-08-31 以降 `/api/v4/*` を Legacy 認定で 403 返却。 当 backend では `safe_fmp_get` の Limit Reach フォールバックで [] を返し silent fail、 数ヶ月気付かず DiagramCard / ProfileCard で segment 空表示。

**対策**:
- 重要 API path は **四半期ごとに probe script で health check** (memory: feedback_supabase_grant_bug.md と同じ pattern)
- `/api/v3/*` `/api/v4/*` から `/stable/*` への path 移行を全 backend で grep
- 既存 移行: `get_segment_data` (v97 で完了)、 未確認: 他 v4 endpoint 残存可能性

### C. AI scoring の session 内 drift bias

3 run mean でも variance ±4 残る。 「初回甘め、 2 回目以降 厳しめ」 pattern で baseline 値が swing。

**対策**:
- anchor 画像 (50 点 / 80 点 / 100 点 サンプル) を rubric に同梱
- run ごとに fresh API session (multi-session pool)
- median + IQR で外れ値除外

### D. CLS 防止は conditional render 撤去 + skeleton 常時 mount

`{data && (<Component data={data}>)}` パターンは data 取得まで height 0 → 後追い mount で縦幅突然増加。 click target が押し下げられて user 誤 click。

**対策**:
- 常時 mount + 内部で 3 state (loading→skeleton / no-data→placeholder / ok→通常)
- skeleton に minHeight を明示
- Web Vitals CLS スコア 0.1 以下を目標

### E. Sprint 連続追加で sanitize 漏れ pattern

Sprint H6 で schema に competitive_moat を追加したが、 ProfileCard の `sanitizeSummaryData` (BLOCKLIST_REGEX Layer 3) で sections 再構築時に moat が落とされていた。 LLM 出力は正しく moat を返していたが frontend で消えていた = 表示されない silent bug。

**対策**:
- LLM tool schema 追加時の checklist: backend `_call_llm` return mapping / frontend sanitize / frontend表示 component の 3 箇所を grep で同時 update
- 既存 v96 教訓 B「LLM endpoint への structural 変更 は個別 commit + smoke test」 を強化
