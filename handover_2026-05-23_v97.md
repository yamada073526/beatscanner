## 4. 永続教訓 (v97 で確立、 memory anchor 候補)

### A. Vision-eval scorer の frame count 依存 bias [feedback_vision_eval_frame_count.md]
G-4 で対策: accordion fallback chain (会社概要 → 最新ニュース → 市場の声 → 8Q履歴 → アナリスト視点) + scroll padding で 8 frame 保証。 ただし scoring 基準 bias は別途 anchor 画像で対策必要。

### B. FMP API path Legacy 認定の silent fail [feedback_fmp_api_legacy.md]
v3/v4 endpoint は 2025-08-31 以降 Legacy 認定で 403、 silent fail 数ヶ月気づかず。 v97 で `/stable/*` 移行済 (segment endpoint)、 他 v3/v4 endpoint も grep audit 推奨。

### C. CLS 防止は「上方 section の高さ envelope」 が最優先
user dogfood で「Pane 3 開くのが面倒」 は scroll 中 CLS が主因。 大型 chart / image 系 section に必ず minHeight envelope を設定、 loading でも data でも同じ height で固定。 vision-eval 静止画では検知できない pattern。

### D. AI scoring の session 内 drift bias
3 run mean でも variance ±4 残る。 「初回甘め、 2 回目以降 厳しめ」 pattern。 anchor 画像 base + fresh session pool で対策 (次セッション着手)。

### E. Sprint 連続追加で sanitize 漏れ pattern
LLM tool schema 拡張時、 backend return mapping / frontend sanitize / 表示 component の 3 箇所同時 update 必要。 v97 で Sprint H6 残債として ProfileCard の sanitize 漏れ発覚。

### F. FMP endpoint 移行時、 response 形式も変わる pattern [feedback_fmp_response_format.md]
`/api/v3/stock-peers` (旧: `{symbol, peersList:[]}`) → `/stable/stock-peers` (新: `[{symbol, name, ...}]`)。 path 変えただけでなく schema も変わる。 path 移行と同時に parsing 全 audit が必要、 silent fail で competition Tab が 「peers 取得失敗」 になる事故あり。

### G. FMP data の部分確定問題 [feedback_fmp_partial_quarter.md]
NVDA Q1 FY2027 のように最新四半期で 1 segment のみ確定 (Data Center) + 残 segment 未報告のケース。 最新 entry をそのまま使うと「100% Data Center」 誤表示。 build 側で「2+ segments 揃った最新四半期」 を選択ロジック必須。

### H. vision-eval anchor rubric で「絶対基準 PDCA」 確立 [feedback_vision_eval_anchor_rubric.md]
旧 rubric は「品質高いか」 主観評価で session 内 drift bias (3 run mean でも variance ±4)。 新 rubric (v97 G-4) は各軸 0/50/80/100 の絶対 anchor を明示 (typography 0=1 font hierarchy 皆無 / 80=4+ size + tabular-nums / 100=serif/sans 混在 + 32px+ 大型 display)。 これにより「過去比較」 ではなく「anchor 自体に対する position」 で判定、 数値の上下が **意味ある signal** になる。 ただし baseline 数値は旧 70 → 新 65 と低下、 直接比較不可。 改善目標 = anchor 80 達成 (各軸で具体的不足点 が note で明示)。

### I. 「不明」 と「ガイダンス非公開」 の区別 [feedback_unknown_vs_not_provided.md]
「verdict: unknown」 が「データ取得失敗 (バグ)」 と「企業が意図的に非開示」 の両方を包含していた。 user 「バグのように見える」 dogfood で判明。 後者は EyeOff icon + 「ガイダンス非公開」 + 「当四半期は業績見通し未発表」 caption で「見えない = 存在しない」 を直感伝達。 前者は「データ取得失敗」 と区別。 LLM 判定 / data 不在 / 企業非開示 の 3 ケースは UI で別表現が必要。
